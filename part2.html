💭 Por que a IA esquece tudo?

Cada conversa com uma IA começa do zero. Sem memória, o modelo não sabe quem você é, qual é o seu projeto, nem quais decisões já foram tomadas. O resultado? Repetição, contexto perdido e respostas genéricas.

// Conversa 5, mesmo projeto, mesma pessoa

Você: Preciso adicionar autenticação ao projeto

IA: Claro! Qual linguagem você usa? Qual framework? Já tem um banco de dados? Usa JWT ou sessões? (pergunta tudo de novo)

~2.000 tokens gastos reconstruindo contexto que já existia

// Conversa 5, com memória ativa

Você: Preciso adicionar autenticação ao projeto

IA: Baseado no seu projeto Node.js + Express com PostgreSQL, recomendo JWT com refresh tokens. Vou seguir o padrão de middleware que você já usa em src/middleware/. (contexto preservado)

0 tokens gastos com re-contexto. Resposta direta e personalizada.

Tokens gastos com contexto por conversa

Sem memória, cada conversa reconstrói o contexto do zero. Com memória, o investimento cai progressivamente.

🗂️ Tipos de Memória em Ferramentas de IA

Diferentes ferramentas implementam memória de formas diferentes. Entender os tipos ajuda a escolher a abordagem certa.

💬

Memória de Sessão

Existe apenas durante a conversa atual. É o padrão em ChatGPT, Claude.ai e todos os chats de IA. Quando a conversa fecha, o contexto desaparece.

Escopo: 1 conversa ChatGPT, Claude.ai, Gemini
👤

Memória do Usuário

Persiste entre conversas. Armazena preferências, papel profissional e padrões de trabalho. ChatGPT Memory e Claude Code auto-memory usam este tipo.

Escopo: cross-session ChatGPT Memory, Claude Code
📁

Memória de Projeto

Vinculada a um repositório ou workspace específico. Inclui decisões de arquitetura, bugs conhecidos e estado do projeto. CLAUDE.md e MEMORY.md são exemplos.

Escopo: 1 repositório Claude Code, Cursor, Windsurf
📋

Instruções Persistentes

Arquivos de configuração que a IA lê automaticamente em cada sessão. São a forma mais previsível de "memória", controlada diretamente pelo dev.

Escopo: configurável CLAUDE.md, .cursorrules, Custom Instructions

Insight: A combinação mais poderosa é instruções persistentes (previsível, controlada) + memória de projeto (dinâmica, aprendida). É exatamente o que o Claude Code faz com CLAUDE.md + auto-memory.

📄 O Sistema MEMORY.md

O Claude Code possui um sistema de memória automática baseado em arquivos. Cada projeto pode ter seu próprio diretório de memória em .claude/memory/.

# Estrutura do diretório de memória

.claude/

  memory/

    MEMORY.md          # Índice: ponteiros para cada arquivo de memória

    user_role.md        # Quem é o usuário, preferências

    feedback_testing.md # Correções e confirmações

    project_api.md      # Decisões de projeto em andamento

    reference_jira.md   # Onde encontrar info externa

Exemplo: MEMORY.md (o índice)

# Memory Index

- [User: Senior Backend Dev](user_role.md) · Node.js, PostgreSQL, prefere testes de integração
- [Feedback: Sem mocks](feedback_testing.md) · Testes devem bater no banco real
- [Projeto: API v3](project_api.md) · Migração em andamento, deadline 15/04
- [Ref: Board Jira](reference_jira.md) · Bugs tracked em PROJ-backend

Exemplo: arquivo de memória individual

---
name: Sem mocks nos testes
description: Testes de integração devem usar banco real
type: feedback
---

Não usar mocks de banco nos testes de integração.

**Why:** Incidente Q3: testes com mock passavam mas a migração
quebrou em produção porque o schema real era diferente.

**How to apply:** Quando criar testes que envolvem banco de dados,
sempre usar um container PostgreSQL de teste, nunca jest.mock().

Ponto-chave: MEMORY.md é apenas o índice. Cada memória vive em seu próprio arquivo com frontmatter YAML. O Claude lê o índice no início de cada conversa e acessa os arquivos individuais conforme necessário.

🏷️ Os 4 Tipos de Memória no Claude Code

O Claude Code organiza memórias em 4 categorias. Cada tipo tem um propósito específico e guia quando e como salvar informações.

👤

User Memory

Quando salvar: ao aprender detalhes sobre o papel, preferências ou conhecimento do usuário

Informações sobre quem você é, o que faz e como prefere trabalhar. Permite que o Claude adapte respostas ao seu nível e contexto profissional. O Claude cria um arquivo .md em .claude/memory/ e adiciona uma entrada no índice MEMORY.md.

Exemplo de trigger:

"Sou data scientist, estou investigando os logs do sistema"

Arquivo criado: user_role.md

---
name: Data scientist focado em observabilidade
description: Usuário é data scientist, investigando logs do sistema
type: user
---

Usuário é data scientist com foco atual em observabilidade e logging.
Adaptar explicações para perfil analítico, usar analogias com
dados, pipelines e métricas quando possível.
🔄

Feedback Memory

Quando salvar: quando o usuário corrige OU confirma uma abordagem

Correções e confirmações sobre como trabalhar. O tipo mais importante: evita que o mesmo erro se repita em conversas futuras. Cria um arquivo .md com a regra, o motivo (Why) e como aplicar (How to apply).

Correção:

"Não use mock no banco. Fomos queimados no trimestre passado quando testes mockados passaram mas a migração falhou"

Arquivo criado: feedback_no_mocks.md

---
name: Sem mocks nos testes de integração
description: Testes de integração devem usar banco real, nunca mocks
type: feedback
---

Não usar mocks de banco nos testes de integração.

**Why:** Incidente Q3: testes com mock passavam mas a migração
quebrou em produção porque o schema real era diferente.

**How to apply:** Quando criar testes que envolvem banco,
sempre usar container PostgreSQL de teste, nunca jest.mock().
📁

Project Memory

Quando salvar: ao aprender sobre trabalho em andamento, metas, decisões ou prazos

Informações sobre o estado atual do projeto que não são deriváveis do código ou histórico git. Inclui decisões, motivações e prazos. Cria um arquivo .md com o fato, o motivo (Why) e como aplicar (How to apply). Datas relativas são convertidas para absolutas.

Exemplo de trigger:

"Estamos congelando merges não-críticos depois de quinta. Time mobile vai cortar a release branch"

Arquivo criado: project_merge_freeze.md

---
name: Merge freeze para release mobile
description: Congelamento de merges não-críticos a partir de 2026-03-05
type: project
---

Merge freeze começa 2026-03-05 para corte de release mobile.

**Why:** Time mobile precisa de branch estável para o release.
Merges não-críticos podem introduzir regressões.

**How to apply:** Sinalizar qualquer PR não-crítico agendado
após 2026-03-05. Sugerir adiar para após o release cut.
🔗

Reference Memory

Quando salvar: ao aprender onde informações vivem em sistemas externos

Cria um arquivo .md com ponteiros para sistemas externos. Funciona como um caderno de endereços: não armazena o conteúdo do sistema externo, apenas onde encontrá-lo. Útil porque o conteúdo muda frequentemente, mas o endereço permanece estável.

Exemplo de trigger:

"Bugs de pipeline são rastreados no projeto Linear 'INGEST'"

Arquivo criado: reference_linear_ingest.md

---
name: Pipeline bugs no Linear
description: Bugs de pipeline são rastreados no Linear projeto INGEST
type: reference
---

Bugs de pipeline → Linear, projeto "INGEST"

Quando o usuário mencionar problemas de pipeline ou bugs de
ingestão, consultar este projeto no Linear para contexto.

⚡ Salvando & Recuperando Memórias

No Claude Code, memórias podem ser salvas explicitamente ou detectadas automaticamente. Veja como usar na prática.

1

Pedido explícito

Diga ao Claude para lembrar algo e ele salva automaticamente no diretório de memória.

"Lembre que neste projeto usamos PostgreSQL com Prisma e testes rodam contra um container Docker local."

2

Detecção automática

O Claude detecta correções, confirmações e informações importantes durante a conversa e salva proativamente.

"Não, não use npm. Este projeto usa pnpm com workspaces."

→ Salva feedback: "projeto usa pnpm com workspaces, não npm"

3

Revisar e editar

Memórias são arquivos Markdown comuns em .claude/memory/. Você pode editar, remover ou reorganizar a qualquer momento.

# Para ver o que o Claude lembra:

"O que você lembra sobre este projeto?"

# Para remover:

"Esqueça a informação sobre o deadline da API v3"

4

Recuperação automática

O MEMORY.md é carregado automaticamente no início de cada conversa. Memórias relevantes são acessadas conforme necessário durante a sessão.

Importante: O Claude verifica se memórias ainda são válidas antes de agir nelas. Se uma memória referencia um arquivo que não existe mais, ele atualiza ou remove automaticamente.

O que NÃO salvar em memória

✕ Padrões de código ou arquitetura (leia do código)
✕ Histórico git ou quem mudou o quê (use git log)
✕ Soluções de debugging (o fix está no commit)
✕ Qualquer coisa já documentada no CLAUDE.md

⚠️ Limites & Cuidados

Memória de IA não é mágica. Entender os limites evita surpresas e decisões baseadas em informação obsoleta.

Memórias ficam obsoletas

Uma memória que diz "usamos Express" pode estar errada se o time migrou para Fastify. Memórias são snapshots de um momento, não verdades permanentes. O Claude tenta verificar antes de agir, mas nem sempre consegue detectar mudanças silenciosas.

Cuidado com informações sensíveis

Memórias são arquivos em texto plano no repositório. Nunca salve senhas, tokens, chaves de API ou PII (dados pessoais identificáveis) como memória. Trate o diretório de memória como qualquer outro arquivo do projeto.

Não confie cegamente

Uma memória diz que a função X existe, mas ela pode ter sido renomeada ou removida. Antes de recomendar algo baseado em memória, o Claude deve verificar no código atual. "A memória diz que X existe" não é o mesmo que "X existe agora".

Curadoria regular

Revise as memórias periodicamente. Remova o que ficou obsoleto, atualize prazos que passaram e consolide memórias fragmentadas. Uma memória limpa é melhor que uma memória grande.

Considere o .gitignore

Se o diretório de memória contém informações pessoais ou específicas do seu workflow, considere adicionar .claude/memory/ ao .gitignore. Memória de projeto pode ser commitada; memória pessoal geralmente não.

🗂️ Organização por Pastas: Skill + Agent

O padrão flat (tudo em .claude/memory/) funciona para projetos pequenos. Mas quando as memórias crescem, o MEMORY.md estoura o limite de 200 linhas e é truncado. O Claude perde acesso a parte das memórias. A solução: subpastas por categoria com Skill + Agent para manter a convenção.

# Flat: tudo no mesmo nível

.claude/memory/

  MEMORY.md                ← índice (200 linhas max!)

  user_role.md

  feedback_no_mocks.md

  feedback_pnpm.md

  project_api_v3.md

  project_merge_freeze.md

  reference_linear.md

  reference_grafana.md

  ... ← 30+ arquivos = MEMORY.md cresce demais

# Subpastas: organizado por domínio

.claude/memory/

  MEMORY.md                ← só ~10 linhas (ponteiros para pastas)

  api/

    README.md              ← índice desta categoria

    auth_flow.md

    rate_limits.md

  ui/

    README.md

    design_system.md

  infra/

    README.md

    deploy_pipeline.md

  ← MEMORY.md leve, Claude lê só a pasta relevante

As 3 peças necessárias

O Claude Code não cria subpastas sozinho: ele precisa de instruções explícitas. A combinação de CLAUDE.md + Skill + Agent resolve isso.

1

CLAUDE.md: a instrução base

Adicione no CLAUDE.md do projeto a regra de como salvar memórias. Isso é lido em toda conversa.

## Memória do Projeto
Memórias ficam em subpastas de `.claude/memory/` por domínio.
Cada pasta tem um README.md como índice local.
Use a skill `/save-memory` para salvar no local correto.
Nunca salve memórias diretamente na raiz de `.claude/memory/`.
2

Skill: o comando de salvar

Uma skill em .claude/commands/save-memory.md que o Claude invoca para criar o arquivo no lugar certo.

.claude/commands/save-memory.md 
---
description: Salva uma memória na subpasta correta
---

Salve uma memória seguindo estas regras:

1. Identifique a **categoria** (api, ui, infra, team, etc.)
2. Crie a pasta `.claude/memory/{categoria}/` se não existir
3. Crie o arquivo `.md` com frontmatter (name, description, type)
4. Atualize `.claude/memory/{categoria}/README.md` com a nova entrada
5. Atualize `.claude/memory/MEMORY.md` se a categoria for nova

Formato do MEMORY.md (índice raiz):
```
# Memory Index
- [API](api/README.md) · autenticação, rate limits, endpoints
- [UI](ui/README.md) · design system, componentes, acessibilidade
- [Infra](infra/README.md) · deploy, CI/CD, monitoramento
```

Formato do README.md de cada pasta:
```
# API Memories
- [Auth Flow](auth_flow.md) · JWT + refresh tokens, middleware pattern
- [Rate Limits](rate_limits.md) · 100 req/min por tenant
```
3

Agent: o curador automático

Um agente em .claude/agents/memory-curator.md que revisa, limpa e reorganiza memórias periodicamente.

.claude/agents/memory-curator.md 
---
description: Revisa e organiza memórias do projeto
allowed-tools: Read, Write, Edit, Glob, Grep
---

Você é o curador de memórias do projeto.

## Tarefas
1. Leia todos os arquivos em `.claude/memory/` recursivamente
2. Para cada memória, verifique:
   - O arquivo/função referenciado ainda existe no código?
   - A data mencionada já passou? (memória expirada)
   - Existe duplicata em outra categoria?
3. Remova memórias obsoletas
4. Consolide duplicatas em uma única memória
5. Atualize todos os README.md de cada subpasta
6. Atualize o MEMORY.md raiz

## Regras
- Nunca remova memórias do tipo `feedback` sem confirmação
- Converta datas relativas para absolutas
- Mantenha MEMORY.md abaixo de 50 linhas
- Cada README.md de subpasta: máx 30 linhas

Por que funciona

MEMORY.md leve: Só ponteiros para pastas (~10 linhas). Nunca atinge o limite de 200 linhas.

Leitura sob demanda: O Claude só abre o README.md da pasta relevante, não carrega todas as memórias.

Escalável: 100+ memórias organizadas em ~8 pastas, cada uma com índice próprio.

Curadoria automática: O agent revisa periodicamente sem intervenção manual.

Nota: Este padrão é uma convenção manual. O Claude Code não cria subpastas por padrão. Você precisa das 3 peças (instrução no CLAUDE.md + skill + agent) para que o Claude siga a convenção. Sem elas, ele volta ao comportamento flat padrão.

🚀 Como começar hoje

Quatro passos para ativar e usar memória de IA no seu projeto.

1

Verifique se a memória está ativa

No Claude Code, verifique se existe o diretório .claude/memory/ no seu projeto. Se não existir, peça: "Lembre que este é um projeto Node.js com TypeScript".

2

Comece com 2-3 memórias fundamentais

Salve seu papel (user), uma preferência forte do time (feedback) e um contexto de projeto atual (project). Não tente salvar tudo de uma vez.

3

Observe a detecção automática

Nas próximas conversas, observe quando o Claude salva memórias por conta própria. Revise o MEMORY.md após algumas sessões para verificar qualidade.

4

Estabeleça um ciclo de curadoria

Uma vez por semana, abra o MEMORY.md e pergunte: cada memória aqui ainda é verdade? Remova o que ficou obsoleto, atualize o que mudou.
