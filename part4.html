🔄 O Ciclo Autônomo de Correção

O objetivo dos Claude Hooks é criar um sistema que se auto-valida. A IA não apenas escreve o código, mas ela é forçada a testá-lo e corrigi-lo antes de te entregar o resultado final. Clique nos estágios para entender o fluxo.

🔌 Tipos de Hooks Disponíveis

Hooks não são apenas para escrita de arquivos. O Claude Code expõe 4 eventos do ciclo de vida do agente que você pode interceptar. Cada um tem um propósito diferente.

Evento

PreToolUse

🛡️

Roda antes de qualquer ferramenta ser executada pelo agente. Pode bloquear ou modificar a ação antes que aconteça.

Casos de uso:

  • • Validar se o arquivo está na lista de permissões
  • • Bloquear escrita em diretórios protegidos
  • • Logar auditoria de toda ação do agente
Evento

PostToolUse

⚙️

Roda depois que uma ferramenta completa. Recebe o output completo da operação. É o hook mais usado para lint e testes.

Casos de uso:

  • • Rodar ESLint/Prettier após escrita de arquivo
  • • Executar testes dos arquivos modificados
  • • Atualizar índices ou geração de tipos
Evento

Stop

🏁

Roda quando o Claude finaliza a resposta completa ao usuário. Ideal para ações de encerramento e notificações.

Casos de uso:

  • • Notificar via Slack quando uma task termina
  • • Gerar relatório de alterações feitas
  • • Rodar um build final de validação
Evento

Notification

🔔

Roda quando o Claude emite um alerta ou notificação interna, como avisos de permissão ou status de longa operação.

Casos de uso:

  • • Customizar alertas de permissão para o time
  • • Integrar com ferramentas de observabilidade
  • • Notificações de desktop em tarefas longas

🎯 Como o Matcher funciona

Cada hook usa um matcher para filtrar em qual ferramenta ele deve disparar. Sem matcher, dispara em todas.

matcher: "Write"

Só dispara quando o agente cria um arquivo novo.

matcher: "Write|Edit"

Dispara em criação e edição de arquivos.

matcher: "Bash"

Dispara antes de qualquer comando de terminal.

🛠️ Implementação Técnica

A configuração vive em ~/.claude/settings.json (global) ou .claude/settings.json (por projeto). Este arquivo conecta o agente ao seu ambiente de desenvolvimento.

📄 .claude/settings.json estrutura real 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit|MultiEdit",
        "hooks": [
          {
            "type": "command",
            "command": "cd $CLAUDE_PROJECT_DIR && npm run lint --fix"
          },
          {
            "type": "command",
            "command": "cd $CLAUDE_PROJECT_DIR && npx vitest run --reporter=verbose"
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "notify-send 'Claude Code' 'Task finalizada!'"
          }
        ]
      }
    ]
  }
}

📦 Variáveis de ambiente disponíveis nos hooks

$CLAUDE_PROJECT_DIR Raiz do projeto atual
$CLAUDE_TOOL_NAME Nome da ferramenta que disparou o hook
$CLAUDE_FILE_PATHS Arquivos afetados pela operação (PostToolUse)
$CLAUDE_SESSION_ID ID único da sessão corrente

Por que Hooks e não CI/CD?

O CI/CD roda depois do push. O Hook roda durante a geração do código. A IA recebe o erro do terminal em tempo real e não entrega o resultado até o comando retornar sucesso.

Global vs. Por Projeto

Hooks em ~/.claude/settings.json valem para todos os projetos. Hooks em .claude/settings.json são específicos do repositório e podem ser versionados com o time.

⚠️ Limites & Cuidados

Hooks são poderosos, mas mal configurados podem travar seu workflow ou criar falsos positivos. Conheça os riscos antes de colocar em produção.

🔴 Riscos a evitar

∞

Loop Infinito

Se o hook sempre retorna falha (ex: lint com regra impossível de satisfazer), o Claude fica em loop tentando corrigir indefinidamente. Sempre teste o hook manualmente antes de ativar.

✓ Defina um timeout máximo no comando: timeout 30 npm run lint

🔓

Permissões de Segurança

Hooks executam com suas permissões de sistema. Um hook mal escrito ou injetado pode executar comandos arbitrários. Nunca copie configurações de hooks de fontes desconhecidas sem revisar.

✓ Prefira hooks com escopo mínimo necessário

🟡 Pontos de atenção

🐢

Performance em projetos grandes

Rodar toda a suíte de testes a cada PostToolUse em um monorepo com 10k testes vai paralisar seu fluxo. Use o matcher para limitar o escopo e filtre apenas os testes relacionados ao arquivo modificado.

🌐

Dependências externas

Hooks que chamam APIs externas, bancos de dados ou serviços de rede podem falhar por razões fora do seu controle (timeout, sem internet). Isso vai bloquear o Claude mesmo que o código esteja correto. Isole hooks de validação de código dos hooks de integração.

🤖

A IA não substitui julgamento humano

Hooks validam o que é verificável por máquina: sintaxe, tipos, cobertura. Eles não detectam lógica de negócio errada, edge cases não cobertos por testes, ou decisões arquiteturais ruins. O código revisado pela IA ainda precisa de olho humano, só que em um nível mais alto.

✅ Boas práticas

1.

Comece com um único hook simples. Adicione complexidade gradualmente.

2.

Sempre use timeout em comandos que podem travar.

3.

Versione o .claude/settings.json no repositório do projeto.

4.

Documente no README o que cada hook faz e por quê existe.

📊 Impacto no Gargalo

A economia de tempo não vem da escrita, mas da revisão. Quando a IA se auto-corrige, o tempo humano é liberado para decisões arquiteturais.

Revisão Manual (Antes)

"Faltou um ponto e vírgula", "Esse nome de variável está ruim", "Esqueceu de tratar o catch".

Revisão com Hooks (Depois)

"Essa abstração faz sentido para o negócio?", "O design da API é escalável?".

Dados simulam a transição de esforço após implementação de hooks. O tempo de revisão manual cai progressivamente conforme a confiança na suíte de testes aumenta.

🔍 De Revisor de Sintaxe a Arquiteto

Como fica o seu dia a dia após desentupir o gargalo? Sua sênioridade agora é aplicada onde a IA ainda falha: Intencionalidade e Negócio.

❌ Review "Old School"

1h Verificando indentação e linter.
2h Apontando erros de lógica básica.
30m Fadigado para o que importa.

✅ Review "AI-Augmented"

0m Linter e testes já passaram nos Hooks.
15m Validando contrato da API e segurança.
2h Focando em Arquitetura e Negócio.

O Mantra da Parte 4:

"A máquina valida o que é verificável. Você julga o que é intencional."

🚀 Como começar hoje

Não precisa configurar tudo de uma vez. O caminho mais seguro é começar simples, validar, e expandir. Aqui está um roteiro de 4 passos.

1

Crie o arquivo de configuração

Crie ou edite o arquivo de hooks do seu projeto:

mkdir -p .claude
touch .claude/settings.json
2

Adicione um hook de lint como primeiro teste

Comece apenas com lint pois é rápido, seguro e dá feedback imediato:

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "cd $CLAUDE_PROJECT_DIR && timeout 20 npm run lint"
          }
        ]
      }
    ]
  }
}
3

Valide em uma task real

Peça ao Claude para criar ou modificar um arquivo e observe o hook disparar no terminal. Confirme que:

  • → O lint rodou automaticamente após a escrita
  • → Um erro de lint forçou o Claude a tentar novamente
  • → O resultado final entregue já estava limpo
4

Expanda com testes e versione com o time

Quando lint estiver funcionando bem, adicione testes. Depois faça commit do .claude/settings.json para que todo o time se beneficie:

git add .claude/settings.json
git commit -m "feat: add Claude Code hooks for lint and tests"
