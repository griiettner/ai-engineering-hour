🔗 O Sistema Integrado

Nas sessões anteriores, estudamos cada ferramenta isoladamente: Hooks, Memory, Agents, Skills, Commands, Plans e MCP. Agora vamos ver o que acontece quando elas trabalham juntas. Não é um "triângulo de ouro", é um sistema completo onde cada peça amplifica as outras.

📦 Ferramentas Isoladas

  • •Hook roda um lint e mostra o resultado
  • •Memory guarda preferências do usuário
  • •Agente executa uma tarefa e encerra
  • •Skill formata um comando reutilizável
  • •MCP conecta a uma fonte de dados

🔗 Ferramentas Integradas

  • •Hook detecta erro → aciona agente → agente corrige → hook valida de novo
  • •Skill coordena 3 agentes em paralelo com papéis diferentes
  • •Agente consulta MCP, gera um Plan, e salva decisões na Memory
  • •Memory informa o agente sobre padrões do projeto antes de cada tarefa
  • •Command aciona um pipeline completo: plan → code → review → commit
💡

O Princípio da Composição

Cada ferramenta do Claude Code foi projetada como uma primitiva composável. Hooks são event listeners. Skills são prompts empacotados. Agents são processos isolados. Memory é estado persistente. MCP é I/O externo. Quando você conecta essas primitivas, o poder não é aditivo: é multiplicativo.

🗺️ Mapa das Ferramentas

Antes de integrar, é preciso entender o papel de cada peça no sistema. Cada ferramenta tem um domínio específico e interfaces claras com as outras.

⚡ Hooks Eventos
🧠 Memory Estado
🤖 Agents Execução
🎯 Skills Reutilização
⌨️ Commands Interface
📐 Plans Estratégia
🔌 MCP I/O Externo
📋 CLAUDE.md Diretrizes

Quem se conecta com quem

⚡ Hooks →

Podem acionar scripts que disparam agentes, atualizam memory, consultam MCP servers, ou executam validações externas. São o "event bus" do sistema.

🤖 Agents →

Podem chamar skills, criar plans, ler/escrever memory, usar MCP tools, e spawnar sub-agentes. São os "workers" que executam qualquer combinação de ferramentas.

🎯 Skills →

Podem ser invocados por agentes, por commands, ou até por outros skills. Definem o "como fazer" reutilizável. Podem coordenar multi-agentes internamente.

🧠 Memory →

É consultada automaticamente no início de cada conversa. Agents e hooks podem atualizar memories. É o "estado de longo prazo" que sobrevive entre sessões.

🧩

Modelo Mental: O Fluxo

CLAUDE.md define as regras. Memory traz o contexto histórico. Um Command ou Skill inicia a ação. Agents executam o trabalho. Hooks validam cada passo. Plans documentam a estratégia. MCP conecta ao mundo externo. Tudo numa cadeia que se auto-reforça.

🎯 Cenários de Integração

Cada cenário mostra ferramentas diferentes trabalhando juntas. Clique nos tabs para explorar as combinações.

Cenário 1

Hook detecta padrão → Agent corrige → Memory registra

Um PostToolUse hook detecta que um arquivo TypeScript foi escrito sem type exports. Um script aciona o Claude que lê o erro e corrige. A memory registra a decisão para que o Claude já saiba o padrão na próxima sessão.

⚡PostToolUse
→
🔍Detecta falta
→
🤖Claude corrige
→
🧠Memory salva 
// .claude/settings.json · hook que valida exports
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/scripts/check-ts-exports.sh \"$TOOL_INPUT\""
          }
        ]
      }
    ]
  }
}

// Fluxo completo:
// 1. Claude escreve um arquivo .ts via Write
// 2. Hook roda check-ts-exports.sh automaticamente
// 3. Script retorna exit 1 + stderr se faltar exports
// 4. Claude lê o erro e adiciona os exports faltantes
// 5. Hook roda de novo → exit 0 → aprovado
// 6. Claude salva na memory: "Este projeto exige type exports"

# .claude/commands/review-pr.md
---
description: Review de PR com agentes especializados
allowed-tools: Agent, Read, Grep, Glob
---

Analise o PR atual com 3 subagentes em paralelo:

1. **Agente de Segurança** (subagent_type: Explore)
   - Busque vulnerabilidades OWASP top 10
   - Verifique inputs não sanitizados
   - Ferramentas: Grep, Read

2. **Agente de Performance** (subagent_type: Explore)
   - Identifique N+1 queries, loops desnecessários
   - Verifique uso de índices em queries SQL
   - Ferramentas: Grep, Read

3. **Agente de Testes** (subagent_type: general-purpose)
   - Verifique cobertura dos paths alterados
   - Sugira testes faltantes
   - Ferramentas: Grep, Read, Bash

Consolide os 3 reports num único sumário com severidade
(critical / warning / info) para cada item encontrado.

Agent Explore → Gera Plan → Agent Executor implementa

Um agente do tipo Plan analisa o codebase, identifica os arquivos relevantes e cria um plano detalhado. Um segundo agente, com contexto limpo, recebe o plano e implementa. O clássico "Plan Then Build" mas agora orquestrado com agentes tipados.

# Passo 1: Agente planejador (subagent_type: Plan)
"Analise a codebase e crie um plano detalhado para
 migrar o auth de JWT para session-based.
 Identifique todos os arquivos afetados,
 a ordem de alteração e possíveis breaking changes."

# O plano é salvo em .claude/plans/auth-migration.md

# Passo 2: Agente executor (subagent_type: general-purpose)
"Implemente o plano em .claude/plans/auth-migration.md
 Siga a ordem exata. Rode os testes após cada arquivo.
 NÃO altere arquivos fora do escopo do plano."

# O executor começa com contexto limpo:
# - Sem o histórico de exploração do planejador
# - Focado exclusivamente na implementação
# - Hooks validam cada escrita automaticamente

# .claude/commands/implement.md
---
description: Pipeline completo de implementação
allowed-tools: Agent, Read, Write, Edit, Bash, Grep, Glob
---

## Pipeline de Implementação

### Fase 1: Contexto
- Leia as memories do projeto para entender decisões anteriores
- Verifique o CLAUDE.md para convenções de código

### Fase 2: Planejamento
- Use um Agent (subagent_type: Plan) para criar o plano
- Salve em .claude/plans/ com nome descritivo

### Fase 3: Implementação
- Use um Agent (subagent_type: general-purpose) para implementar
- O agent segue o plano, os hooks validam cada escrita

### Fase 4: Review
- Spawne um Agent (subagent_type: Explore) para review
- Se encontrar problemas, volte à Fase 3 com as correções

### Fase 5: Finalização
- Atualize a memory com decisões tomadas
- Crie o commit com mensagem baseada no plano
- Resuma o que foi feito para o humano validar

📋 Receitas Avançadas

Configurações prontas para copiar e adaptar. Cada receita combina pelo menos 3 ferramentas diferentes.

Receita

Hook de Auto-Correção + Memory de Padrões

O hook detecta erros de lint após cada Write/Edit. O Claude lê o erro e corrige. Se o mesmo tipo de erro aparece repetidamente na sessão, salva uma memory de feedback para evitar o padrão nas próximas sessões.

⚡ Hook + 🤖 Auto-correção + 🧠 Memory 
// .claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [{
          "type": "command",
          "command": "npx eslint --no-warn-ignored \"$TOOL_INPUT\" 2>&1 | head -20"
        }]
      }
    ]
  }
}

// O Claude aprende com o padrão de erros:
// Sessão 1: corrige 5x "missing semicolons"
// → salva memory de feedback: "usar sempre semicolons"
// Sessão 2: nunca mais gera código sem semicolons
Receita

Agent Orquestrador que reutiliza Skills

Um agente customizado usa a tool Skill internamente para invocar skills existentes como /commit ou /review-pr. O agente decide qual skill chamar baseado no contexto, sem que você precise orquestrar manualmente.

🤖 Agent + 🎯 Skill + ⌨️ Command

Você é um orquestrador de tarefas de desenvolvimento.

## Workflow
1. Analise a tarefa recebida
2. Decida quais skills são necessários
3. Execute-os na ordem correta

Se o review encontrar problemas, corrija antes de commitar.

MCP + Plan + Memory para Sprint Planning

Um command consulta o tracker de tarefas via MCP, cruza com o estado da codebase, gera um plan de trabalho, e salva na memory o contexto da sprint para que todas as sessões subsequentes tenham consciência do que está em andamento.

🔌 MCP + 📐 Plan + 🧠 Memory 
# .claude/commands/sprint-plan.md
---
description: Gera plan da sprint a partir do tracker
allowed-tools: Agent, Read, Write, Grep, Glob
---

## Sprint Planning Automatizado

1. Use a MCP tool do tracker para buscar tickets
   com status IN_PROGRESS e TODO

2. Para cada ticket, use um Agent (subagent_type: Explore)
   para mapear os arquivos afetados na codebase

3. Gere um plan em `.claude/plans/sprint-YYYY-MM-DD.md`
   com: ticket, arquivos, dependências, ordem de execução

4. Salve uma memory do tipo "project" com:
   - Tickets da sprint e seus status
   - Decisões de priorização
   - Bloqueios identificados

5. Apresente um resumo para o humano validar
Receita

CLAUDE.md que evolui via Hook + Memory

Um hook no evento Stop analisa os feedbacks da sessão. Se padrões recorrentes são detectados, sugere atualizações ao CLAUDE.md. Assim as regras do projeto evoluem organicamente baseadas no uso real, mas sempre com aprovação humana.

⚡ Hook (Stop) + 🧠 Memory + 📋 CLAUDE.md 
// Hook no evento Stop
{
  "hooks": {
    "Stop": [
      {
        "hooks": [{
          "type": "command",
          "command": "echo 'Revise as memories de feedback desta sessão. Se algum padrão apareceu 3+ vezes, sugira uma adição ao CLAUDE.md.'"
        }]
      }
    ]
  }
}

// Ciclo de evolução:
// 1. Você dá feedback durante a sessão
// 2. Claude salva como memory de feedback
// 3. No Stop, o hook sugere: "Adicionar ao CLAUDE.md?"
// 4. Você aprova ou rejeita a alteração
// 5. Regra agora vale para TODAS as sessões futuras

O efeito composto

Cada ferramenta integrada reduz o atrito da seguinte. Memory reduz o onboarding de cada sessão. Hooks eliminam verificações manuais. Plans garantem que o agente não desvie do escopo. O ganho não é linear: é composto. 3 ferramentas integradas valem mais que 5 ferramentas isoladas.

⚠️ Limites & Cuidados

Integração adiciona poder, mas também complexidade. Nem toda combinação faz sentido. Saiba quando simplificar.

❌ Integração por integração

Conectar 6 ferramentas para uma tarefa que um agente único resolve em 30 segundos. O overhead de coordenação supera qualquer ganho. Se um agente único resolve, não escale.

✅ Integre quando o problema exigir

A integração deve surgir de uma necessidade real: erros recorrentes justificam hooks, perda de contexto entre sessões justifica memory, tarefas complexas justificam plan + agents. Comece simples, adicione peças conforme o atrito aparece.

❌ Memory como banco de dados

Salvar tudo na memory: logs, outputs de agentes, histórico completo de conversas. Memory tem limite de linhas no index. Guarde apenas o que o Claude precisa saber na próxima sessão e não consegue derivar do código ou git.

✅ Memory seletiva e atualizada

Salve decisões, preferências e contexto que não está no código. Atualize ou remova memories obsoletas. Uma memory precisa e enxuta vale mais que centenas de registros desatualizados.

❌ Hooks que fazem tudo automaticamente

Um hook que detecta, corrige, commita e pusha sem intervenção humana. Se algo der errado, o dano já foi feito antes de você perceber. Hooks devem validar e reportar, não tomar ações irreversíveis sozinhos.

✅ Hooks validam, humanos aprovam

O hook verifica e reporta. O Claude propõe a correção. Você aceita ou rejeita. Ações irreversíveis (push, deploy, delete) sempre passam pelo humano. O sistema sugere, mas não decide sozinho.

🧠

A regra dos 3 minutos

Se você não consegue explicar como as ferramentas se conectam no seu setup em 3 minutos, provavelmente tem complexidade desnecessária. Um setup integrado deve ser compreensível. Se nem você entende o fluxo, como vai debugar quando algo quebrar?
