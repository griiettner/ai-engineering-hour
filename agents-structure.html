Mapa da documentação

O foco aqui passou a ser o scaffold de referência e como ele mantém compatibilidade entre ferramentas.

Repo de referência

`agents-scaffolding` organiza um setup canônico e compatível entre ferramentas

O ponto de partida recomendado agora é o repositório griiettner/agents-scaffolding. Ele separa bootstrap, memória, skills, tasks e tooling de sync, mantendo .agents/ como estrutura canônica e adicionando compatibilidade explícita para Claude Code e Codex.

Princípio do scaffold

A estrutura de verdade vive em `.agents/`. Integrações específicas de ferramenta são camadas de compatibilidade, não a fonte canônica.

O que o scaffold acerta

  • Mantém `AGENT.md` curto e orientado a roteamento
  • Separa `.agents/memory/`, `.agents/skills/` e `.agents/tasks/`
  • Expõe skills canônicas para Claude via `.claude/skills/` gerado
  • Usa frontmatter e `index.yaml` para descoberta barata

O que ele evita

  • Prompt gigante como memória total do projeto
  • Mistura de workflow, decisão durável e histórico de task
  • Acoplamento do layout a uma única ferramenta
  • Índices manuais que envelhecem rápido

Exemplo real de bootstrap leve

Trecho alinhado ao `AGENT.md` do scaffold, com rotas por contexto e leitura mínima.

# Contexto do Agente

Este arquivo é o contexto inicial leve para agentes.
Use-o como mapa de rotas, não como memória completa do projeto.

- Trabalho de task ou ticket: leia `.agents/tasks/index.yaml` primeiro
- Regras de workflow de tickets: leia `.agents/skills/tkt-management/SKILL.md`
- Roteador de memória durável: leia `.agents/memory/MEMORY.md`
- Contexto de arquitetura: leia `.agents/memory/architecture/index.yaml` primeiro
- Decisões duráveis: leia `.agents/memory/decisions/index.yaml` primeiro

Carregue apenas os arquivos relevantes para a tarefa atual.

Canônico + adapters

AGENT.md e .agents/ são a base. CLAUDE.md e .claude/skills/ existem para compatibilidade, não para duplicar a fonte de verdade.

Pilares reais

Memory guarda conhecimento durável. Skills guardam procedimento reutilizável. Tasks guardam a história operacional. Tools regeneram índices e bridges.

Compatibilidade

Skills continuam canônicas em `.agents/skills/`, com exposição por ferramenta

O scaffold fixa a regra importante: a skill vive em `.agents/skills/<name>/SKILL.md`. Claude recebe stubs gerados em `.claude/skills/`. Codex pode usar `agents/openai.yaml` ao lado da skill. Cursor fica no nível de convenção de naming e invocation.

`tkt-management/`

Define o lifecycle canônico de tasks, modos slim/full e completion gates.

`tkt-plan/` + `tkt-exec/`

Separam planejamento de execução e preservam contexto útil em `report.md` ou `updates/`.

`tkt-sync/` + `skill-new/`

Atualizam metadados gerados e criam novas skills mantendo a estrutura consistente.

Ponto estrutural

Não edite a cópia exposta para a ferramenta. Edite a skill canônica em `.agents/skills/` e regenere os adapters.

Estrutura pequena, canônica e compatível

.agents/skills/tkt-plan/
├─ SKILL.md
└─ agents/
   └─ openai.yaml

.claude/skills/tkt-plan/
└─ SKILL.md   # stub gerado apontando para a skill canônica

# SKILL.md
Crie ou atualize `plan.md` apenas quando a task tiver decisões de implementação em aberto.

- Use o contexto do ticket em `.agents/tasks/TKT-XXX/`
- Evite planejar quando `task.md` já for diretamente executável
- Preserve apenas o plano mínimo útil

Memória durável

Memory segue o mesmo princípio: índice primeiro, arquivo mínimo depois

AGENT.md roteia para .agents/memory/MEMORY.md, que por sua vez roteia para categorias como arquitetura, decisões e princípios. O scaffold recomenda frontmatter pequeno, índices temáticos e leitura estritamente seletiva.

Regras de memória

  • Arquivos duráveis começam com YAML frontmatter.
  • O subset suportado deve permanecer simples e previsível.
  • Leia `index.yaml` antes do Markdown detalhado.
  • O arquivo canônico continua sendo o `.md`, não o índice.

Fluxo de leitura

`AGENT.md` → `MEMORY.md` → `index.yaml` temático → arquivo certo. O objetivo não é documentar tudo; é permitir que a ferramenta ache só o contexto que precisa.

Exemplo de frontmatter de memória

---
id: architecture-overview
title: Visão geral da arquitetura
type: architecture
status: ativo
created: 2026-05-25
updated: 2026-05-25
tags:
  - arquitetura
  - roteamento
read_when:
  - alterando a estrutura de pastas
---

Work history

Tasks no scaffold são a loja canônica de work-unit history

O repo de referência usa uma pasta por ticket ou work item. Há modo slim para chores simples e modo full para trabalho com conceito, plano, execução e relatório. O importante é manter a história operacional dentro da própria task.

Fonte da verdade

A pasta `TKT-XXX/` é a história real do trabalho. O scaffold demonstra `task.md` como mínimo e usa `concept.md`, `plan.md`, `report.md` e `updates/` só quando o tipo de tarefa justificar.

Papel do índice

`tasks/index.yaml` e os shards em `tasks/indexes/` ajudam a escolher o ticket certo antes de abrir a pasta. Eles continuam sendo metadado derivado.

# TKT-001 Tarefa

Automação

Índices só funcionam bem quando são regeneráveis e validados

O scaffold assume `index.yaml` como atalho de descoberta, mas também assume scripts de sync e validação para evitar drift. É isso que mantém a árvore barata de ler sem virar documentação quebrada.

Regra operacional

Se o índice é importante para a ferramenta, ele precisa ser gerado por tooling do repositório.

Índice de memória por categoria

entries:
  - id: principles-constitution
    file: .agents/memory/principles/constitution.md
    status: ativo
  - id: architecture-overview
    file: .agents/memory/architecture/overview.md
    status: ativo

Shard de tasks por intervalo

- range: TKT-001-025
  file: .agents/tasks/indexes/TKT-001-025.yaml
  count: 2
  updated: 2026-05-25
  statuses:
    - concluido: 1
    - em_andamento: 1

Esquema estável

Defina campos YAML consistentes para memory, tasks e índices.

Sync e validação

Use `tkt_sync.py`, `tkt_validate.py` e `sync_claude_skills.py` para manter tasks, índices e compatibilidade alinhados.

Descoberta barata

Leia o menor arquivo útil primeiro. O índice continua existindo só para reduzir custo de contexto.

Erros comuns

A maior parte das falhas vem de duplicação e acoplamento por ferramenta

O scaffold existe justamente para separar canônico de compatibilidade. Quando isso quebra, a manutenção sobe e a descoberta piora rápido.

Transformar `AGENT.md` em dump file

Se o bootstrap deixa de rotear e tenta ensinar tudo, o custo de entrada explode.

Editar stubs gerados em vez da skill canônica

Isso cria drift entre `.agents/skills/` e `.claude/skills/` e destrói a regra de fonte única.

Misturar memória durável com histórico de task

Conhecimento de projeto e narrativa operacional têm ciclos de vida diferentes e precisam de lugares diferentes.

Assumir auto-discovery igual em todas as ferramentas

Claude, Codex e Cursor têm capacidades diferentes. O scaffold lida com isso explicitamente em vez de fingir simetria.

Manter índices e bridges sem sync

Se `index.yaml` e stubs não acompanham a árvore, eles viram ruído e não roteadores.

Ferramentas

O mesmo scaffold suporta ferramentas diferentes, com integrações diferentes

A parte estável é a organização do repositório. O que muda por ferramenta é descoberta de skills, sintaxe de invocação e nível de automação nativa.

Claude Code

Bridge explícita

Use `.claude/skills/` como stub gerado e mantenha `CLAUDE.md` alinhado para redirecionar à estrutura canônica.

Codex

Metadata local

Mantenha `agents/openai.yaml` pequeno e ao lado da skill canônica quando quiser display metadata e invocation policy.

Cursor

Convenção, não bridge

Use os mesmos nomes de skill e argumentos, mas trate a integração como convenção até existir suporte nativo ou adapter seu.

Bootstrap

O caminho recomendado agora é clonar `griiettner/agents-scaffolding`

Em vez de reconstruir essa estrutura manualmente, use o scaffold como base e substitua o conteúdo genérico pelas regras reais do seu projeto.

Clone direto

git clone https://github.com/griiettner/agents-scaffolding
cd agents-scaffolding
1

Clone o scaffold

Use griiettner/agents-scaffolding como base em vez de recriar a árvore do zero.

2

Troque os placeholders

Atualize `AGENT.md`, `CLAUDE.md`, arquitetura, princípios, decisões e exemplos de task para refletir o projeto real.

3

Defina suas skills canônicas

Mantenha cada workflow em `.agents/skills/`, com `agents/openai.yaml` apenas quando houver necessidade real no Codex.

4

Regenere adapters e índices

Rode `sync_claude_skills.py`, `tkt_sync.py` e `tkt_validate.py` sempre que a estrutura relevante mudar.

5

Mantenha o subset de frontmatter simples

O scaffold assume YAML pequeno e previsível para evitar parse ambíguo e índices quebrados.

6

Abra uma sessão nova no repo clonado

Depois do bootstrap, carregue a ferramenta a partir da raiz do repositório para que a descoberta local de contexto e skills funcione corretamente.

Resumo final

Clone o scaffold, trate `.agents/` como sistema de registro, exponha compatibilidade por ferramenta só onde necessário e deixe índices e stubs sempre regeneráveis.
