🪨 O Problema da Memória Sempre Carregada

Na sessão 2 você viu os quatro tipos de memória. Funciona. Até o dia em que o projeto tem 200 arquivos curados, e a IA carrega tudo antes da primeira pergunta. Memória estática tem um teto, e ele chega rápido.

📦 Memória Estática · Load All

A IA lê todos os .md no startup. Funciona para 10 arquivos. Em 200, o contexto fica saturado antes de você digitar a primeira pergunta. Toda chamada paga o preço de carregar o que provavelmente não vai usar.

~18.000 tokens · todo turno

🎯 Memória RAG · Query on Demand

A IA não carrega memória. Ela consulta. AGENTS.md fica em ~500 tokens e responde "vou buscar". O resto chega chunk a chunk, só quando relevante. Startup constante. Contexto enxuto.

~600 tokens · startup · queries sob demanda

🔍 RAG em 90 Segundos

RAG significa Retrieval-Augmented Generation. A IA não responde com o que sabe de cor. Ela busca antes, depois responde. Não é mágica, é índice de biblioteca turbinado. Três operações, três conceitos. Em noventa segundos você tem o suficiente para conversar.

🔢 Embed

Texto vira vetor. Cada chunk de texto recebe coordenadas num espaço de muitas dimensões.

"autenticação" e "login" ficam vizinhos. "autenticação" e "café" ficam longe.

🗂️ Index

Vetor e texto vão para um banco com BM25 (busca lexical) e índice vetorial (busca semântica).

Os dois índices coexistem no mesmo SQLite. Sem servidor, sem nuvem.

🎯 Search

Pergunta vira vetor. Banco devolve os chunks mais próximos. LLM pequeno reordena por relevância real.

Top-3 ou top-5 entram no contexto. O resto fica de fora.

O espaço vetorial em ação

Cada ponto é um chunk indexado. Quando a query chega, o pipeline dispara feixes em busca dos vizinhos mais próximos. Os três que iluminam são o top-K que entra no contexto.

A query "como autenticar usuário?" não procura a palavra. Procura o vizinho. Por isso "login", "auth" e "credentials" iluminam mesmo sem aparecer no texto da pergunta.

💡

Três acertos, três responsabilidades

BM25 acerta termos. Vetor acerta intenção. Rerank acerta prioridade. Sozinhos cada um falha em casos óbvios. Juntos cobrem a maior parte do espaço de busca real. Essa combinação é o que separa um RAG decente de um RAG que te decepciona depois de duas semanas.

Espera, o que é "embedding" mesmo?

Pense num mapa onde palavras são pontos. Palavras com significado parecido ficam perto, palavras sem relação ficam longe. Um embedding é a coordenada do ponto. Modelos como embeddinggemma transformam qualquer texto numa coordenada de algumas centenas de dimensões. Você não precisa entender as dimensões, só saber que distância no espaço significa semelhança no significado.

📚 QMD · Seu RAG Local

QMD é o motor de busca local criado por Tobias Lütke. Roda 100% offline depois do setup. Hybrid search com BM25, vetor e rerank, tudo dentro de um único arquivo SQLite. Sem servidor, sem nuvem, sem API key. Você dá a ele uma pasta de markdown e ele te devolve um índice consultável.

Embedding
embeddinggemma 300M
~300MB · embedding-only · não generativo
Busca Lexical
SQLite FTS5
termos exatos, idiomas, prefixos
Busca Vetorial
sqlite-vec
vizinhos próximos no espaço semântico
Rerank
qwen3-reranker 0.6B
LLM pequeno reordena top-30

Onde mora o índice

~/.cache/qmd/index.sqlite

# Tudo num arquivo só.
# 10MB para projeto pequeno.
# 30 a 50MB para repositórios grandes com docs + código.
# Backup é cp. Reset é rm.

Smart chunking, não corte burro

QMD não corta texto a cada N caracteres. Ele pontua os boundaries: heading vale 50 a 100, code block 80, parágrafo 20. Tenta chunks de ~900 tokens com 15% de overlap. Para código, usa tree-sitter (TypeScript, Python, Go, Rust) e respeita fronteiras de função.

Chunker burro
Corta no meio da frase. Embedding fica diluído. Retrieval traz pedaços inúteis.
Smart chunking
Corta em heading ou fim de bloco. Cada chunk fica coerente. Embedding representa um conceito real.

Os comandos que você vai usar de verdade:

# Registra a pasta de memória como uma coleção
qmd collection add agent-memory ~/.agents/memory --mask "**/*.md"

# Gera embeddings (uma vez, depois é incremental)
qmd embed agent-memory

# Reindex manual (quando você edita .md fora do fluxo normal)
qmd embed agent-memory --force

A primeira vez baixa os modelos (~300MB embedding + ~600MB reranker). Da segunda em diante é instantâneo.

# Busca lexical pura (rápida, para termos exatos)
qmd search agent-memory "timestamp utc"

# Busca vetorial pura (semântica)
qmd vsearch agent-memory "como serializar datas"

# Busca híbrida + rerank (o caminho recomendado)
qmd query agent-memory "convenção de datas no projeto" --top 5

# JSON estruturado para consumo por agente
qmd query agent-memory "auth refresh flow" --json -n 3

90% do uso prático é qmd query. Os outros dois servem para tunar o pipeline ou debugar resultado ruim.

📁 Memória Indexável · O .agents/memory/

Na sessão 6 você viu o .claude/ como workspace seguro. Funciona, mas trava o conhecimento dentro do Claude. Trocar para Cursor ou Codex é perder tudo. A solução é tirar a memória do silo do tool: ela mora em .agents/, que é tool-agnostic. Claude Code, Cursor, Codex, todos apontam para o mesmo lugar.

Formato obrigatório · YAML frontmatter + dense bullets

QMD chunkeia esses arquivos, então a regra é simples: todo bullet precisa fazer sentido sozinho como fato pesquisável. Prose é veneno. Cada token tem que carregar sinal.

---
topic: timestamps
last_updated: 2026-05-19
tags: [date, serialization, utc, postgres]
---

# Timestamp Conventions

## Storage
- All DB timestamps are TIMESTAMPTZ in PostgreSQL
- Application code never writes naive datetimes
- Convert to UTC at the API boundary, not in the DB layer

## Serialization
- API responses use ISO-8601 with explicit Z suffix
- Frontend converts to user TZ at render time only
- Logs always print UTC, never local

## Gotchas
- node-postgres returns timestamps as Date objects with local TZ
- pg-protocol-v2 fixes this; v1 needs manual handling
- Snake-case columns in PG, camelCase in app, mapped in repo layer

Cada bullet acima cabe num único chunk se necessário. Cada um responde uma busca diferente.

❌

Prose-heavy (não indexável)

No nosso projeto a gente costuma
guardar timestamps em UTC porque
isso evita problemas de timezone
quando a gente faz deploys em
diferentes regiões. Acho que isso
foi decidido em algum momento mas

42 palavras para dizer "UTC sempre". O chunker corta no meio, o embedding dilui, a busca falha.

✅

Dense bullets (indexável)

- All timestamps stored in UTC
- Decision: avoid TZ drift across
  multi-region deploys
- Convert to local TZ only at
  render time on frontend

17 palavras, 3 fatos pesquisáveis. Cada um vira chunk útil. Cada um responde uma pergunta diferente.

🛑

Regra de ouro · query first, write second

Antes de criar um arquivo novo, sempre rode qmd query agent-memory "<topic>". Se já existe algo cobrindo o tema, atualize o arquivo existente. Memória duplicada degrada o rerank: a IA fica entre dois chunks contraditórios e responde mal os dois.

🪤 O Teto da Memória Estática

Memória estática resolve o problema de carregamento. Não resolve o problema de captura. Você ainda escreve cada .md à mão. E aqui aparece o verdadeiro teto: a maior parte do conhecimento que sua sessão gera nunca é registrado, porque você está ocupado fazendo o trabalho, não documentando.

O que acontece numa sessão real

  • →Você descobre que o pg-protocol-v1 ignora TZ no insert
  • →Aplica fix, testa, mergeia, fecha o ticket
  • →Decide com o time: refresh tokens sempre rotacionam, sem grace period
  • →Faz refactor, percebe que connection pooling deveria ser global
  • →Fim do dia: cabeça cheia, ninguém escreveu nada em .md

Resultado em 3 meses

  • ✗Outro dev cai no mesmo bug do pg-protocol-v1
  • ✗A decisão sobre refresh tokens vira "alguém lembra por que isso?"
  • ✗Connection pooling continua local em três serviços diferentes
  • ✗O git log tem a verdade mas ninguém vai ler 240 commits
  • ✗A IA não tem como saber. Você não escreveu.

E pior: troque de tool e perca tudo de novo

# Você passou 6 meses curando .cursor/rules/...
$ cd projeto-x
$ codex
→ Codex não lê .cursor/rules
→ Suas regras curadas viram contexto morto
→ Você reescreve no formato do Codex

Conhecimento preso em formato proprietário é conhecimento condenado a virar custo recorrente.

Captura manual vs captura automática · por tipo de lição

Workaround temporário fica baixo de propósito. Não é tudo que vira lição.

♻️ Reflect · Memória que se Captura Sozinha

Reflect é uma skill (gancho com a Sessão 7) que roda no fim de qualquer sessão significativa, em qualquer tool. Em uns dois minutos ela faz a leitura do que aconteceu, decide o que vale guardar, propõe candidatos e atualiza a memória. Você não para o flow. Você só revisa os candidatos depois.

1
Lê o contexto da sessão
.agents/state/STATE.md + git log -20 + git status --short
↓
2
Identifica 1 a 3 lições reais
regra confirmada por evidência · padrão observado · decisão arquitetural
↓
3
Roteia por confidence
alta → learn.py (graduate imediato) · média → stage.py (review queue)
↓
4
Escreve reflection file

↓
5
Reindexa o QMD
próxima sessão já consulta o conhecimento novo no first query

Como fica um reflection file de verdade:

Exemplo de uma sessão real (anonimizada) de hotfix em refresh token. Menos de 20 linhas. Sem prose. Cada bullet é fato auditável contra commit ou teste.

---
date: 2026-05-19
session: jwt-rotation-bug
tags: [auth, jwt, refresh, bug]
---

# JWT Rotation Hotfix

## What changed
- Fixed refresh token race in POST /auth/refresh
- Added DB-level lock on user_sessions row during rotation

## Decisions made
- Refresh always rotates; no grace period for the old token
- 5-second clock skew tolerance documented in adr/004

## Lessons staged
- "Refresh endpoints must hold row-level lock during rotation" → staged
- "Clock skew below 10s breaks under our deploy latency" → graduated

A automação está em capturar e propor. O julgamento continua humano: quais lições viram regra permanente, quais ficam em quarentena, quais são descartadas. A próxima seção mostra esse ciclo de governança em detalhe.

🔄 Ciclo de Vida das Lições

Nem toda lição vira regra. Nem toda lição é descartada. Reflect e auto-learn produzem candidatos. O lifecycle separa o joio do trigo com uma regra simples: confidence routing. Alta confiança grada direto, média entra na fila, baixa some. E cada graduate precisa de rationale, sem exceção.

Auto-learn · o daemon que estuda enquanto você trabalha

launchd dispara dev.brain.auto-learn a cada 30 minutos. O daemon varre git activity, chama Gemma 4 local (Ollama, localhost:11434), extrai 0 a 3 lições e roteia. Se Ollama não está rodando, ele falha silenciosamente. Nunca bloqueia o seu commit.

Curadoria periódica · o memory-steward

Captura automática é só metade do ciclo. Sem revisão periódica, a memória acumula duplicatas com palavras diferentes, lições obsoletas que ninguém rejeita e contradições que o rerank tenta conciliar e falha. O memory-steward é um agente manual que você invoca de tempos em tempos para fazer essa limpeza. Ele não escreve memória nova; ele audita o que já existe.

1 · Audit
Lista candidates pendentes via list_candidates.py. Roda show.py para ver dashboard de stale e graduated por idade.
2 · Flag
Identifica duplicatas (qmd query no claim), lições contraditórias (mesma área, decisões opostas) e memória stale (last_updated > 90 dias e zero hit em recall).
3 · Decide
graduate.py com rationale para promover, reject.py --reason para arquivar duplicata, edição manual para realinhar contexto contraditório. Reindex no fim.
# Ciclo típico de curadoria (semanal ou quando recall fica ruim)
$ python3 .agents/tools/brain/list_candidates.py --priority high
$ python3 .agents/tools/brain/show.py --stale-days 90

# Rejeitar duplicata identificada
$ python3 .agents/tools/brain/reject.py L-2026-04-12-a \
    --reason "Duplicata de L-2026-03-08-c, mesmo claim sobre rate limits"

# Promover candidate validado
$ python3 .agents/tools/brain/graduate.py L-2026-05-01-b \
    --rationale "Confirmado em PR #312 e tests/auth/test_refresh.py"

# Reindex para o qmd refletir as mudanças
$ .agents/engine/bin/qmd-reindex.sh

Cadência sugerida: rode quando recall.py começar a retornar matches contraditórios, ou semanalmente em projetos ativos. O sinal mais claro é quando o agente busca e cita duas lições que se contradizem. É a memória pedindo ajuda.

📌

--rationale não é rubber-stamp

"Evidence holds" não é rationale. "PR #234 corrige o bug, teste em tests/auth/test_refresh.py passa" é. O sistema obriga texto livre justamente para forçar evidência citável. Quem grada hoje precisa explicar para quem vai consultar amanhã.

⚠️ Anti-patterns da Memória Viva

Memória viva é frágil. Cinco erros fazem o sistema degradar em semanas. Quatro são fáceis de cair sem perceber. Conhecer cada um economiza meses de busca dando resposta errada.

❌ 1. Duplicar memória sem buscar antes

Você cria project/timestamps.md sem rodar qmd search. Mês depois, alguém criou project/date-conventions.md. Ambos dizem coisas levemente diferentes.

Consequência: rerank fica confuso. Top-3 vem com chunks contraditórios. A IA responde mal.

✅ Query first, write second

Antes de criar arquivo novo, sempre qmd query agent-memory "<topic>". Se já existe algo, atualize o existente.

Reuse > create. Atualize com last_updated. Mantenha um arquivo por domínio.

❌ 2. Memória prose-heavy

Você escreve parágrafos contando a história da decisão. Fica bonito para humano, péssimo para chunker. Boundary cai no meio da frase, embedding fica diluído.

Consequência: busca semântica falha. O chunk retornado não responde a pergunta sozinho.

✅ 1 bullet = 1 fato

Cada bullet precisa fazer sentido sozinho. Sem "isso", "aquilo", "como falei acima". Cada linha é uma busca em potencial.

Teste mental: se esse bullet vier sozinho num search, faz sentido? Se não, reescreva.

❌ 3. Graduate sem evidência

Auto-learn enfileira 8 candidatos. Você grada todos com --rationale "evidence holds". Mês depois, a IA recall lições falsas que viram regra.

Consequência: o sistema confia em alucinações destiladas. Cada erro vira lei.

✅ Rationale com link auditável

Todo graduate cita PR, commit ou arquivo de teste. "PR #234, tests/auth/test_refresh.py passa". Se você não consegue citar, rejeite ou stage.

Aceitar 2 lições por semana com rationale forte vale mais que 20 com "evidence holds".

❌ 4. Lock em diretório de tool

Memória mora em .claude/memory/. Quando o time abre Cursor ou Codex, ninguém acha. Cada tool recria a sua versão. Triplica o trabalho.

Consequência: conhecimento preso por escolha de tool. Switching cost vira proibitivo.

✅ Tool-agnostic em .agents/

Memória em .agents/memory/. .claude/REDIRECT.md, .cursor/rules/project.mdc e MCP do Codex apontam para lá.

Migra de tool sem perder uma linha. O conhecimento é do projeto, não do editor.

❌ 5. Reflect sem dedup

Reflect roda sem consultar reflections recentes. Em duas semanas você tem 5 reflection files dizendo "refresh tokens devem rotacionar" com palavras ligeiramente diferentes.

Consequência: bloat na memória. Rerank degrada porque embeddings ficam saturados de variações.

✅ Query antes de escrever reflection

Reflect consulta git log curto e roda qmd query no tema. Se já existe lesson cobrindo, atualiza last_updated em vez de criar arquivo novo.

Reflection nova só quando o tema é novo. Refinamento atualiza o existente.

❌ 6. Memória sem curadoria periódica

Reflect e auto-learn escrevem, ninguém revisa. Em três meses a memória tem duplicatas com palavras diferentes, lições obsoletas que contradizem decisões novas e arquivos stale que ninguém mais consulta. O agente busca, recebe matches conflitantes e o rerank tenta conciliar o que não tem conciliação.

Consequência: qualidade do retrieval degrada silenciosamente. Você só percebe quando o agente cita lição obsoleta como verdade atual.

✅ Memory-steward em cadência fixa

Rode o memory-steward semanalmente, ou quando recall.py retornar matches contraditórios. Ciclo curto: list_candidates → flag duplicatas via qmd query → reject.py com --reason ou graduate.py com --rationale → reindex. Trinta minutos a cada duas semanas evitam dívida que leva dias para limpar.

Curadoria não é overhead. É o que separa memória viva de cemitério de notas.

🧭

Padrão dos seis erros: economizar julgamento humano

Todos os seis vêm da mesma raiz: querer pular a etapa de revisão. Duplicar acontece quando você não busca. Prose acontece quando você não edita. Graduate sem evidência acontece quando você bate olho rápido. Lock em tool acontece quando você não pensa em portabilidade. Reflect sem dedup acontece quando você confia 100% na automação. Memória sem curadoria acontece quando você confunde "automatizar captura" com "automatizar julgamento". A memória viva só vive porque o humano ainda decide.

🚀 Próximos Passos · Comece em 4 movimentos

Quatro passos para sair de memória estática para memória viva ainda esta semana. Cada passo é independente, cada um já entrega valor.

1

Instale o QMD

Siga o setup oficial em github.com/tobi/qmd. O repo cuida do download dos modelos (embeddinggemma e reranker) na primeira execução. Confirme com qmd --version.

git clone https://github.com/tobi/qmd.git
cd qmd && ./install.sh
qmd --version

A primeira execução pode levar uns minutos para baixar os modelos. Da segunda em diante é instantâneo.

2

Crie a primeira memória

Mova o que já está em .claude/memory/ ou .cursor/rules/ para .agents/memory/project/. Use o formato YAML frontmatter + dense bullets. Comece com 1 ou 2 arquivos. Não precisa migrar tudo de uma vez.

mkdir -p .agents/memory/project
cat > .agents/memory/project/timestamps.md <<'EOF'
---
topic: timestamps
last_updated: 2026-05-19
tags: [date, utc]
---

# Timestamp Conventions
## Storage
- All DB timestamps are TIMESTAMPTZ
- Convert to UTC at the API boundary
EOF
3

Indexe e teste a busca

Registre a coleção, gere os embeddings, faça a primeira query. Se o top-1 retornar o chunk relevante, você está pronto. Se vier algo estranho, revise o formato do .md.

qmd collection add agent-memory .agents/memory --mask "**/*.md"
qmd embed agent-memory
qmd query agent-memory "timestamp utc" --top 3

Para reindex futuro, escolha um dos dois: qmd embed agent-memory --force direto, ou crie um wrapper .agents/engine/bin/qmd-reindex.sh que também roda triggers do seu fluxo.

4

Ative reflect no fim das sessões

Crie a skill reflect em .agents/skills/reflect/SKILL.md com o protocolo de 5 passos. Rode manualmente no fim das primeiras sessões para calibrar. Depois automatize via post-commit hook ou daemon.

mkdir -p .agents/skills/reflect
mkdir -p .agents/memory/reflections
mkdir -p .agents/tools/brain
# adicione learn.py, stage.py, recall.py, graduate.py
# a partir da documentação .agents/AGENTS.md

Comece manual. Quando estiver confortável com o formato dos reflection files, adicione o daemon auto-learn (Ollama + Gemma 4) para capturar entre sessões.
