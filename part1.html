⚔️ IDE vs. CLI vs. Web

O primeiro passo para aumentar a produtividade com IA é sair do chat do navegador. Cada interface tem um propósito diferente, e usar a errada cria fricção desnecessária e ignora os guardrails que você definiu para o projeto.

💻

IDE

Cursor / VS Code + Copilot

  • ✅ Código gerado já respeita seu padrão
  • ✅ Sem copiar/colar, edita direto no arquivo
  • ✅ Autocomplete ciente de todo o codebase
Use quando: desenvolvimento do dia a dia, features incrementais, edições pontuais.
⌨️

CLI Agente

Claude Code / Gemini CLI

  • ✅ Executa tarefas em lote autonomamente
  • ✅ Roda comandos, testes e linter sozinho
  • ✅ Ideal para refatorações e migrações grandes
Use quando: automação pesada, refatoração de monorepos, tarefas multi-arquivo.
🌐

Chat Web

Claude.ai / ChatGPT

  • ❌ Perda constante de contexto do projeto
  • ❌ Erros de indentação por copiar/colar
  • ❌ Ignora completamente seus Guardrails
Use quando: exploração conceitual, perguntas sem código, brainstorm inicial de arquitetura.
💡

A Regra dos 3 Ambientes

Devs experientes usam os três. Web para pensar, IDE para construir, CLI para automatizar. O erro mais comum é tentar construir código de produção no Web: sem contexto, sem guardrails, sem rastreabilidade.

🧠 Anatomia da IA: Janela de Contexto

Por que a IA "começa a falar bobagem" em conversas longas? Porque ela tem uma memória de curto prazo finita, medida em tokens. Quando esse espaço se esgota, ela descarta as informações mais antigas, incluindo seus guardrails.

O que é um Token?

Pedaços de palavras processados pelo modelo. Em média, 1.000 tokens ≈ 750 palavras. Claude tem uma janela de 200k tokens, mas atenção seletiva importa mais do que tamanho bruto.

A "Memória de Curto Prazo"

Quando projeto + histórico de conversa excedem a janela, a IA começa a descartar as instruções do início, exatamente onde ficam seus guardrails e regras de projeto.

⚠️
O Risco da Distração: Enviar node_modules, dist ou logs gigantes consome tokens valiosos e "distrai" a atenção da IA do que realmente importa.
✅
A Solução: Crie um arquivo .claudeignore (Claude Code) ou .cursorignore (Cursor) para excluir pastas ruidosas do contexto automaticamente.

65% dos tokens gastos com ruído. A IA "vê" mal o código que importa.

🛡️ Guardrails por Ferramenta

Cada ferramenta lê as regras do projeto de um arquivo diferente, mas o objetivo é sempre o mesmo: dar uma bússola à IA para que o código gerado já saia alinhado com os padrões do time.

📄 CLAUDE.md | Claude Code 
# Regras do Engenheiro Sênior: Projeto X

## Perfil
- Você é um Engenheiro de Software Sênior especializado em TypeScript e Clean Architecture.
- Suas respostas devem ser concisas e focadas em código testável.

## Guardrails de Design (Regras de Ouro)
- **Injeção de Dependência:** Nunca instancie classes diretamente; use inversão de controle.
- **Tratamento de Erros:** Todo código assíncrono deve ter try/catch com logging.
- **Tipagem:** Proibido o uso de `any`. Use `unknown` se o tipo for incerto.
- **Testes:** Para cada feature, inclua o esqueleto do teste unitário em Vitest.

## Stack Atual
- Frontend: React 19 + Vite + TailwindCSS
- Backend: Node.js + Fastify + Prisma
- Testes: Vitest + Testing Library
Lido automaticamente pelo Claude Code em cada sessão. Coloque na raiz do projeto e versione com o time.
📄 .cursorrules | Cursor IDE 
# Cursor Rules: Projeto X

You are a Senior TypeScript engineer. Follow these rules strictly:

## Code Style
- Never use `any`. Prefer `unknown` for unsafe types.
- All async functions must have explicit return types.
- Use dependency injection, never instantiate services inline.

## Architecture
- Feature Folders pattern: keep domain logic inside `src/features/`.
- No business logic in controllers. Controllers are thin adapters only.

## Testing
- Always suggest a Vitest skeleton for new functionality.
- Mocks must use `vi.fn()`, never manual class overrides.
O Cursor também suporta regras por pasta em .cursor/rules/*.mdc, útil para regras específicas por domínio.
📄 .github/copilot-instructions.md | GitHub Copilot 
# GitHub Copilot Instructions: Projeto X

## Language and Stack
- TypeScript only. No JavaScript files should be created.
- Use React 19 functional components with hooks, no class components.
- State management: Zustand. Do not suggest Redux or Context API.

## Code Conventions
- Named exports only. No default exports except for page components.
- Error handling: always use Result types (Ok/Err pattern), no raw throws.
- API calls: use the internal `httpClient` wrapper, never raw `fetch`.

## Testing
- Framework: Vitest + @testing-library/react.
- Test files live alongside source: `Button.test.tsx` next to `Button.tsx`.
Funciona com Copilot Chat e autocompletion. Deve estar em .github/copilot-instructions.md na raiz do repositório.
O princípio é o mesmo para todos: definir quem a IA é no seu projeto e como ela deve se comportar. A ferramenta muda, o conceito não. Versione sempre esses arquivos junto com o código.

📂 Planejamento Básico: O Trio Essencial

Antes de pedir à IA para escrever a primeira linha de código, garanta que ela conheça o terreno. Três arquivos simples servem de âncora de contexto, evitando sugestões fora do padrão, versões erradas de libs, e estruturas incompatíveis com o projeto.

DECISIONS.md Avançado

TECH_STACK.md

Lista exata das bibliotecas, frameworks e versões utilizadas.

Objetivo: Evita que a IA sugira React 17 quando você usa 19, ou instale axios quando o projeto já usa ky. 
# Tech Stack: Projeto X

## Frontend
- React 19.0 + TypeScript 5.4
- Vite 5.2 (build) | Vitest 2.0 (testes)
- TailwindCSS 3.4 | Radix UI (componentes)
- Zustand 4.5 (estado global)

## Backend
- Node.js 22 + Fastify 4.x
- Prisma 5.x (ORM) | PostgreSQL 16
- Zod (validação de schema)

ARCHITECTURE.md

Explica a organização estrutural e os padrões de comunicação.

Objetivo: Se diz "Feature Folders", a IA não criará arquivos soltos em /controllers ou /views gigantes. 
# Architecture: Projeto X

## Padrão: Feature Folders
src/
  features/
    auth/
      auth.service.ts   # lógica de negócio
      auth.routes.ts    # controller fino
      auth.schema.ts    # validação Zod
      auth.test.ts      # testes unitários
  shared/
    http/               # wrapper httpClient
    errors/             # Result types

## Regras
- Controllers são adaptadores finos, zero lógica de negócio.
- Serviços recebem dependências via parâmetro (sem instanciar).

REQUIREMENTS.md

Descreve a regra de negócio central, casos de uso e edge cases.

Objetivo: Mantém a IA focada no "o que" estamos construindo. Impede que ignore casos de borda cruciais ou proponha soluções que não atendem o usuário real. 
# Requirements: Feature: Autenticação

## Fluxo Principal
1. Usuário envia email + senha via POST /auth/login
2. Sistema valida credenciais contra hash bcrypt
3. Retorna JWT (access: 15min) + Refresh token (7 dias)

## Edge Cases Críticos
- Após 5 tentativas falhas: bloquear por 15 min (rate limit por IP)
- Token expirado: retornar 401 com código "TOKEN_EXPIRED" (não 403)
- Refresh token rotativo: invalidar o anterior ao emitir novo

## Fora do Escopo (v1)
- OAuth / SSO (será na v2)
- 2FA (será na v2)

🚫 Anti-Patterns Mais Comuns

Saber o que não fazer acelera mais do que qualquer dica de produtividade. Estes são os 4 comportamentos que mais drenam o potencial da IA no dia a dia.

Anti-pattern #1

❌ O Prompt Obra

Jogar todos os requisitos de uma vez esperando que a IA entregue tudo perfeito.

"Crie um sistema de autenticação completo com JWT, refresh token, rate limit, logs de auditoria, 2FA e integração com OAuth do Google."

✅ O Prompt Cirúrgico

Uma tarefa por vez, validando e construindo incrementalmente sobre o que foi aprovado.

"Baseando-se no REQUIREMENTS.md, implemente o endpoint POST /auth/login. Apenas login por email e senha por agora."

Anti-pattern #2

❌ Corrigir Só no Chat

Quando a IA usa axios e você quer ky, você fala no chat, e na próxima sessão ela usa axios de novo.

"Não, use ky ao invés de axios" → Correto apenas nesta mensagem.

✅ Corrigir na Raiz

Vá no CLAUDE.md e adicione a regra. O comportamento muda permanentemente para todo o time.

# Em CLAUDE.md:
- HTTP: use sempre `ky`, nunca `axios` ou `fetch` diretamente. → Correto para sempre.

Anti-pattern #3

❌ Arrastar a Conversa

Continuar uma sessão longa mesmo depois de notar que a IA está esquecendo contexto, gerando inconsistências ou repetindo erros.

Conversa com 80 mensagens, IA ignora as regras do início, sugere padrões que já foram corrigidos antes.

✅ Abrir Nova Sessão

Detecte o sinal de deriva e encerre. Abra uma nova sessão com um prompt de "handoff" resumindo o que foi feito.

"Contexto: implementamos o login e o refresh token. Os testes passam. Próxima tarefa: implementar o rate limit por IP."

Anti-pattern #4

❌ Aceitar Sem Ler

Aprovar o código gerado sem entender a intenção, criando dívida técnica oculta que ninguém consegue manter depois.

"Parece certo, aceitar tudo." → 3 semanas depois ninguém sabe por que aquele código existe.

✅ Revisão de Intenção

Você não precisa verificar cada vírgula (o hook faz isso). Mas precisa entender por que essa solução foi escolhida.

"Explica por que usaste um Map aqui ao invés de um array com find()."

⚙️ O Fluxo de Trabalho (Mão na Massa)

Com ferramentas e arquivos base prontos, este é o ritual para iniciar qualquer tarefa. Seguir este fluxo transforma a dinâmica de "ditador de código" para orquestrador.

1

Indexação da IDE

Ao abrir o projeto, aguarde (ou force) a IDE indexar o repositório. A IA só pode consultar o que ela sabe que existe. Sem indexação, ela opera no escuro.

2

O "Check-in": Primeiro Prompt

Sempre contextualize a IA com os arquivos âncora antes de pedir a tarefa. Não presuma que ela lembrou da sessão anterior.

💬 "Leia os arquivos TECH_STACK.md, ARCHITECTURE.md e CLAUDE.md na raiz. Confirme que entendeu os padrões antes de prosseguirmos."
3

Validação Estrutural

Avalie a resposta. Se ela sugerir algo que viola seu padrão (ex: usou axios quando você usa ky), não corrija apenas no chat.

💡 O Segredo: Vá no CLAUDE.md, adicione a regra restritiva lá, e peça para a IA reler. Você corrige a "raiz" do comportamento, não apenas uma linha temporária.
4

O Loop de Iteração

Após a primeira resposta aprovada, construa incrementalmente. Cada prompt subsequente deve referenciar o que já foi feito.

💬 "O endpoint de login está funcionando e os testes passam. Agora adicione o mecanismo de refresh token conforme o REQUIREMENTS.md."
5

Fechando a Sessão

Sessões longas degradam. Detecte os sinais (IA repetindo erros já corrigidos, ignorando regras, gerando código inconsistente) e encerre. Abra uma nova sessão com um prompt de handoff:

💬 "Retomando contexto: implementamos login + refresh token. Testes passam. Próxima tarefa: rate limit por IP (ver REQUIREMENTS.md, seção Edge Cases)."

🚀 Como começar hoje

Não precisa configurar tudo de uma vez. Quatro ações concretas que você pode fazer ainda hoje para sair do setup zero para um ambiente funcional.

1

Escolha e instale sua ferramenta principal

Para desenvolvimento do dia a dia, escolha uma IDE. Para automações pesadas, instale o Claude Code.

# Instalar Claude Code (CLI agente)
npm install -g @anthropic-ai/claude-code

# Verificar instalação
claude --version
2

Crie o arquivo de guardrails com pelo menos 3 regras

Não espere o arquivo estar perfeito. Comece com o mínimo e evolua conforme encontra comportamentos que precisa corrigir.

# Para Claude Code:
touch CLAUDE.md

# Para Cursor:
touch .cursorrules

# Para Copilot:
mkdir -p .github && touch .github/copilot-instructions.md
3

Crie TECH_STACK.md e ARCHITECTURE.md (mínimo viável)

10 linhas cada é suficiente para começar. O objetivo é que a IA não "adivinhe" sua stack.

git add CLAUDE.md TECH_STACK.md ARCHITECTURE.md .claudeignore
git commit -m "feat: add AI context files for Claude Code"
4

Execute o "Check-in" na sua próxima tarefa real

Use o template do Passo 2 do Fluxo de Trabalho na próxima feature que você for implementar. Observe a diferença de qualidade na primeira resposta.

💬 Template: "Leia TECH_STACK.md, ARCHITECTURE.md e CLAUDE.md. Confirme que entendeu os padrões. Depois, [descreva a tarefa]."
