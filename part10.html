🎭 O Paradoxo do Teste com IA

Pedir para a IA "gerar testes para esta função" produz testes que validam o que o código faz, não o que o código deveria fazer. A cobertura sobe rápido, a confiança não acompanha. O paradoxo está em usar a métrica errada para a pergunta certa.

📈 Cobertura ≠ Confiança

95% de cobertura com mocks vazios e asserções tautológicas. O relatório fica verde. O bug em produção continua existindo. Cobertura mede linhas executadas, não comportamento garantido.

🎯 Tests as Spec

60% de cobertura com testes que falham quando alguém quebra o contrato. Cada teste é uma frase do contrato. Se a frase está vermelha, alguma promessa foi quebrada.

Mesma função, dois estilos de teste:

// IA gera teste depois do código existir
describe('parseDate', () => {
  it('returns a result', () => {
    const result = parseDate('2026-05-12');
    expect(result).toBeDefined();
  });

  it('handles input', () => {
    const result = parseDate('foo');
    expect(result).toBeTruthy();
  });
});

// Cobertura: 100%. Bugs detectados: 0.
// Humano escreve teste antes, define o contrato
describe('parseDate', () => {
  it('parses ISO date 2026-05-12 to Date(2026, 4, 12)', () => {
    expect(parseDate('2026-05-12'))
      .toEqual(new Date(2026, 4, 12));
  });

  it('rejects 2026-02-30 with day-out-of-range', () => {
    expect(() => parseDate('2026-02-30'))
      .toThrow(/day out of range/);
  });

  it('rejects empty string with empty-input', () => {
    expect(() => parseDate(''))
      .toThrow(/empty input/);
  });
});

// Cobertura: 60%. Bugs prevenidos: vários.

Cobertura sobe, regressões continuam

Valores ilustrativos para discussão, não benchmark.

🔄 Inversão Red-Green

O ciclo red-green-refactor clássico assume um humano escrevendo o teste. Quando a IA entra na equação, a ordem importa muito mais. Quem escreve o teste primeiro define o contrato. Quem escreve o código primeiro define o que será testado.

Tradicional
  • 1.Humano escreve teste vermelho
  • 2.Humano implementa o código
  • 3.Humano refatora

Funciona, mas é lento. Bom para projetos críticos.

IA gerando teste depois
  • 1.IA escreve código
  • 2.IA escreve teste verde
  • 3.Coverage sobe, valor fica em zero

Test theater puro: validação que valida a si mesma.

Recomendado
  • 1.Humano escreve teste vermelho (define contrato)
  • 2.IA implementa para passar nos testes
  • 3.Humano revisa o código gerado

Velocidade da IA, garantia do contrato humano.

Três níveis de prompt para gerar testes:

Gere testes para esta função.

A IA inventa cenários baseada no que vê no código. Ela não tem como conhecer o contrato real.

Gere testes para esta função cobrindo: input null, string vazia
e valores inválidos. Use casos de borda explícitos.

Melhor que o anterior, mas ainda só cobre o que a IA enxerga. Sem contrato explícito.

Estou implementando parseDate(input: string). Os seguintes
testes precisam passar (não altere a especificação):

- parseDate('2026-05-12') retorna Date(2026, 4, 12)
- parseDate('2026-02-30') lança InvalidDateError
  com mensagem 'day out of range'
- parseDate('') lança InvalidDateError
  com mensagem 'empty input'
- parseDate('not a date') lança InvalidDateError

Implemente parseDate. Não modifique os testes. Se algum
teste parecer errado, pergunte antes de prosseguir.

Contrato explícito. A IA não pode inventar para fazer passar. Tem que implementar de verdade.

🔗 Pipeline TDD Integrado

Testes não vivem sozinhos. Eles participam do sistema integrado visto na sessão anterior. Hooks rodam testes, agentes corrigem o código quando falham, memórias guardam padrões. O resultado é um ciclo onde a IA propõe, o teste valida, o humano aprova.

1
IA escreve código
Tools Write ou Edit alteram um arquivo do projeto
↓
2
PostToolUse hook roda os testes
Script bash escolhe o runner adequado pelo nome do arquivo
↓
3
Testes falham e devolvem stderr
Exit code 1 e mensagem do runner são lidos pelo Claude
↓
4
Agente corrige o código (não o teste)
A regra é dura: o contrato definido no teste é imutável
↓
5
Hook roda novamente, exit 0
Memory grava o padrão para evitar o mesmo erro depois

Configuração do hook em .claude/settings.json

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/run-tests.sh \"$TOOL_INPUT\""
          }
        ]
      }
    ]
  }
}

Script .claude/hooks/run-tests.sh

#!/usr/bin/env bash
file_path="$1"
case "$file_path" in
  *.test.ts|*.spec.ts) npm test --silent -- "$file_path" ;;
  *_test.py|test_*.py) pytest -q "$file_path" ;;
  *_test.go) go test -run "$file_path" ;;
  *) exit 0 ;;
esac

Memory de feedback em .claude/memory/feedback_tests.md

---
name: feedback-tests
description: Testes deste projeto sempre batem em PostgreSQL real, nunca em mocks

---

**How to apply:** ao escrever testes de camada de

A regra inquebrável

Quando o teste falha, o agente corrige o código, nunca o teste. Se o teste estiver realmente errado, o humano altera no commit seguinte. Misturar essas duas situações gera o pior dos mundos: testes que se acomodam ao bug.

🪞 Mocks vs Realidade

Mockar é uma decisão arquitetural, não um padrão default. Cada nível da pirâmide de testes paga um preço diferente em velocidade e ganha um valor diferente em garantia. O problema do "mocka tudo" é que velocidade vira ilusão de cobertura.

⚠️

A inversão da pirâmide

Times que pedem para a IA "gerar testes" geralmente recebem 95% de unit tests com mocks. A pirâmide inverte: muito mock, pouco real. O resultado é confiança falsa. Force a discussão sobre qual nível cada teste deve estar antes de gerar uma linha sequer.

⚠️ Anti-patterns

Quatro armadilhas aparecem com frequência quando IA escreve testes sem supervisão de contrato. Reconhecê-las antes do code review evita test theater institucional.

1. Tautologia

Errado

expect(true).toBe(true);

Prova nada. Existe só para inflar coverage.

Certo

expect(sum(2, 3)).toBe(5);

Verifica um comportamento específico.

2. Mock-everything

Errado

jest.mock('./db');
jest.mock('./api');
jest.mock('./logger');

Você está testando o mock, não o código.

Certo

// integration test
const db = await startTestcontainerPostgres();
expect(await findUser(db, 42))
  .toEqual({ id: 42 });

Mocka I/O externo só quando inviável testar de verdade.

3. Snapshot-only

Errado

expect(render(<Card/>))
  .toMatchSnapshot();

Snapshot quebra a cada CSS change. Diff é aprovado sem leitura.

Certo

expect(screen.getByRole('button'))
  .toHaveTextContent('Salvar');

Asserção semântica, robusta a mudanças visuais.

4. Happy-path only

Errado

it('parses a date', () => {
  expect(parseDate('2026-05-12'))
    .toBeTruthy();
});

Um teste por função, sempre com input válido. Bordas ficam descobertas.

Certo

it.each([
  ['', /empty/],
  ['2026-02-30', /range/],
  ['foo', /invalid/],
])('rejects %s', (i, m) => {
  expect(() => parseDate(i)).toThrow(m);
});

Tabela de casos de borda explícitos, fácil de estender.

📊 Confiança Real vs Cobertura

Cobertura é uma métrica de implementação: quantas linhas o teste tocou. Bugs evitados é uma métrica de produto: quantos defeitos chegaram ao usuário. Os dois números não estão correlacionados. Escolha qual deles você quer otimizar antes de pedir para a IA gerar uma única linha de teste.

Valores ilustrativos para discussão, não benchmark real.

Cobertura mede o quanto o teste se passeou pelas linhas do código. Bugs evitados mede o quanto o usuário foi protegido. Os dois não estão correlacionados. Escolha a segunda como métrica de discussão em revisões e relatórios.

🥒 Cucumber para Contratos Legíveis

Quando o contrato precisa ser entendido por gente que não escreve código (PM, designer, QA, stakeholder do produto), Gherkin é uma escolha pragmática. O arquivo .feature vira o contrato: linguagem natural que descreve comportamento. A IA lê esse contrato e gera tanto a implementação quanto as step definitions, mas a fonte da verdade fica num arquivo que qualquer pessoa do time consegue revisar.

👥 Por que considerar BDD com Cucumber

  • •PM, designer e QA conseguem ler e contribuir com cenários sem precisar abrir o IDE
  • •A especificação fica versionada junto com o código, na mesma branch e no mesmo PR
  • •A IA recebe um contrato em linguagem natural e gera implementação + step definitions
  • •Reduz ambiguidade: discussões entre tech e produto acontecem em cima do mesmo arquivo
  • •Cenários servem como documentação viva, sempre alinhada com o que o sistema realmente faz

Exemplo: features/parse_date.feature

# language: pt
Funcionalidade: Parsing de datas no formato ISO

  Como desenvolvedor
  Quero converter strings ISO em objetos Date
  Para garantir validação consistente em toda a aplicação

  Cenário: Data válida no formato ISO
    Dado uma string "2026-05-12"
    Quando eu chamar parseDate
    Então devo receber Date(2026, 4, 12)

  Cenário: Data inexistente no calendário
    Dado uma string "2026-02-30"
    Quando eu chamar parseDate
    Então deve lançar InvalidDateError com mensagem "day out of range"

  Cenário: Entrada vazia
    Dado uma string ""
    Quando eu chamar parseDate
    Então deve lançar InvalidDateError com mensagem "empty input"

Esse arquivo é o contrato. PM, QA e dev leem e revisam a mesma fonte da verdade.

Step definitions geradas pela IA: features/steps/parse_date.steps.js

import { Given, When, Then } from '@cucumber/cucumber';
import { strict as assert } from 'node:assert';
import { parseDate } from '../../src/parseDate.js';

Given('uma string {string}', function (input) {
  this.input = input;
});

When('eu chamar parseDate', function () {
  try {
    this.result = parseDate(this.input);
  } catch (e) {
    this.error = e;
  }
});

Then('devo receber Date\\({int}, {int}, {int}\\)', function (y, m, d) {
  assert.deepEqual(this.result, new Date(y, m, d));
});

Then('deve lançar InvalidDateError com mensagem {string}', function (msg) {
  assert.equal(this.error.name, 'InvalidDateError');
  assert.match(this.error.message, new RegExp(msg));
});

A IA gera o glue code a partir do .feature. Você revisa as asserções, não a sintaxe.

✅ Quando vale a pena

  • •Produto define regras de negócio que mudam com frequência
  • •QA participa da escrita de cenários, não só da execução
  • •Compliance, regulação ou auditoria precisa ler a especificação
  • •Equipes distribuídas onde alinhamento assíncrono é crítico

⚠️ Quando o custo supera o ganho

  • •Lógica puramente técnica sem stakeholder não-tech envolvido
  • •Time pequeno onde dev e PM são a mesma pessoa
  • •Step definitions virando salada de regex difícil de manter
  • •Cenários se tornando dump de campos em vez de comportamento real

Setup mínimo (Node.js)

npm install --save-dev @cucumber/cucumber
mkdir -p features/steps
echo 'features/**/*.feature' > cucumber.json

Para Python use behave, para Ruby use cucumber-ruby, para Java use cucumber-jvm. A ideia é a mesma.

💡

O contrato fica fora do código

A grande vantagem não é o Cucumber em si, é onde o contrato vive. Quando ele está em Gherkin, sai do silo dos devs e vira artefato compartilhado. PM revisa, QA expande, dev implementa, IA gera glue. A fonte da verdade deixa de ser o código e passa a ser o comportamento descrito.

🚀 Próximos Passos

Quatro movimentos concretos para começar a tratar testes como contrato no seu projeto.

1

Comece pelo contrato

Antes de pedir código para a IA, escreva os 3 a 5 cenários de borda mais importantes como testes. Esse é o contrato que a IA precisa cumprir.

touch tests/parseDate.spec.ts
2

Configure o hook de teste

Crie .claude/hooks/run-tests.sh e registre como PostToolUse para Write|Edit em .claude/settings.json. O ciclo passa a rodar sozinho.

mkdir -p .claude/hooks && touch .claude/hooks/run-tests.sh
3

Salve padrões na memória

Quando o time decidir "todos os testes de auth usam testcontainers", grave como feedback_tests.md em .claude/memory/. Vira regra automática nas próximas sessões.

touch .claude/memory/feedback_tests.md
4

Audite o test suite existente

Use um agente para procurar tautologias, mock-everything e snapshot-only no seu repositório. O resultado costuma ser doloroso mas necessário.

grep -rn "toMatchSnapshot\|toBeTruthy\|toBeDefined" tests/

Mantra da Parte 10:

"Testes não provam que o código funciona. Especificam o que ele deve fazer."
