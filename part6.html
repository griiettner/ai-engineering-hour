🎯 O que são Skills?

Skills estendem o que o Claude consegue fazer. Você cria um arquivo SKILL.md com instruções e o Claude passa a ter essa ferramenta disponível. Você pode invocar diretamente com /nome-da-skill, mas a parte poderosa é outra: o Claude pode decidir usar a skill sozinho quando percebe que ela é relevante para o que você pediu.

🧑‍💻

Você invoca

Digita /deploy quando quer executar o deploy. Você controla o timing — útil para workflows com efeitos colaterais.

$ /review-pr 142
🤖

O Claude invoca

Você pede "como este código funciona?" e o Claude detecta que a skill explain-code é relevante e a carrega automaticamente.

# A description da skill diz:
"Use quando o usuário perguntar
 como algo funciona."
🗂️

Já existe outro formato: .claude/commands/

Antes das Skills, o Claude Code usava Commands — arquivos .md simples em .claude/commands/. Ambos criam comandos slash e continuam funcionando. Skills são o formato evoluído: suportam frontmatter YAML, invocação automática pelo Claude, arquivos de suporte e execução em subagentes. Veja o comparativo completo na seção .

📁 Onde as Skills vivem

Nível Caminho Escopo
Enterprise managed settings Todos os usuários da org
Pessoal ~/.claude/skills/<nome>/ Todos os seus projetos
Projeto .claude/skills/<nome>/ Apenas este projeto
Plugin <plugin>/skills/<nome>/ Onde o plugin está ativo

Quando skills têm o mesmo nome em níveis diferentes, a prioridade é: enterprise > pessoal > projeto.

⚖️ Commands vs Skills

Ambos criam comandos slash invocáveis com /nome. A diferença está em como são definidos e no que conseguem fazer. Skills são o formato recomendado — Commands continuam funcionando e seguem suportados.

📄

Command (legado)

# .claude/commands/review-pr.md
# (arquivo único, sem frontmatter obrigatório)

Revise o PR $ARGUMENTS verificando:

- Segurança: inputs não validados
- Performance: N+1 queries
- Testes: casos de borda

Responda com Status + lista de Issues.
📁

Skill (recomendado)

# .claude/skills/review-pr/SKILL.md
# (diretório + frontmatter YAML)

---
name: review-pr
description: Revisa PRs. Use ao analisar
  um pull request ou diff.
disable-model-invocation: true
allowed-tools: Bash(gh *), Read
---

Revise o PR $ARGUMENTS verificando:
# ... mesmo conteúdo do command ...

🔄 Migrar ou não migrar?

Commands existentes não precisam ser migrados. Continue usando-os se atendem à necessidade. Migre para Skills quando quiser auto-invocação, arquivos de suporte, ou controle de quando o Claude pode acionar.

⚡ Quando usar cada um

  • Command: instrução simples, sem controle de invocação necessário
  • Skill: quando você quer que o Claude saiba usar por conta própria, ou precisa de frontmatter, arquivos de suporte ou subagente isolado
⚠️

Conflito de nomes: se existir um Command e uma Skill com o mesmo nome, a Skill tem precedência. Evite nomes duplicados entre .claude/commands/ e .claude/skills/.

🔬 Anatomia: SKILL.md

Cada skill é um diretório com um arquivo SKILL.md obrigatório. O arquivo tem duas partes: frontmatter YAML entre --- que configura o comportamento, e o corpo Markdown com as instruções. Clique em cada campo para entender o que faz.

Estrutura do diretório

.claude/skills/
└── review-pr/
    ├── SKILL.md          # obrigatório — instruções e frontmatter
    ├── checklist.md      # opcional — referência carregada quando necessário
    └── scripts/
        └── validate.sh   # opcional — script que o Claude pode executar

.claude/skills/review-pr/SKILL.md

---
name: review-pr
description: Revisa PRs buscando problemas
  de segurança e performance. Use
  quando revisar um pull request.
disable-model-invocation: true
allowed-tools: Bash(gh *), Read
argument-hint: [número-do-pr]
---

Você é um revisor sênior deste projeto.
Analise o PR $ARGUMENTS verificando:

- Segurança: inputs não validados
- Performance: N+1 queries
- Testes: casos de borda descobertos

Para critérios completos, veja
[checklist.md](checklist.md).

## Output Format
**Status**: ✅ Aprovado / ⚠️ Ajustes
**Issues**: ordenados por severidade

👆 Clique em qualquer linha do frontmatter para entender o que ela controla.

📥 Variáveis de Substituição

$ARGUMENTS

Tudo que foi passado após o nome da skill.

/fix-issue 123 → $ARGUMENTS = 123

$ARGUMENTS[0], $0

Argumento por posição (índice 0).

/migrate SearchBar React Vue → $0 = SearchBar

${CLAUDE_SKILL_DIR}

Diretório da skill. Útil para referenciar scripts bundled.

${CLAUDE_SESSION_ID}

ID da sessão atual. Útil para logs por sessão.

📦 Skills Nativas do Claude Code

Skills nativas já vêm com o Claude Code e estão disponíveis em qualquer sessão. Diferente dos comandos built-in que executam lógica fixa, essas skills são orientadas por prompt — o Claude orquestra o trabalho usando suas ferramentas, podendo criar agentes paralelos e adaptar a execução ao seu projeto.

/batch Agente Paralelo
▼

Orquestra mudanças em larga escala no codebase em paralelo. Decompõe em 5–30 unidades independentes e abre um PR por unidade.

/batch migrate src/ de Solid para React

Pesquisa o codebase, apresenta o plano para aprovação, e só então spawna os agentes. Requer repositório git.

/simplify 3 Agentes em Paralelo
▼

Revisa arquivos alterados recentemente em busca de problemas de reuso, qualidade e eficiência — e os corrige.

/simplify focus on memory efficiency

Spawna 3 agentes de review em paralelo, agrega os achados e aplica as correções. Pode receber foco opcional.

/loop Polling
▼

Executa um prompt repetidamente em um intervalo enquanto a sessão estiver aberta.

/loop 5m verificar se o deploy terminou

Útil para monitorar deploys, babysit de PRs, ou re-executar outra skill periodicamente.

/debug Diagnóstico
▼

Ativa logging de debug para a sessão atual e analisa o log para diagnosticar problemas.

/debug o Claude parou de responder corretamente

Debug está desligado por padrão. Executar /debug no meio de uma sessão começa a capturar logs a partir daquele ponto.

/claude-api Auto-invocada
▼

Carrega referência da API Anthropic e Agent SDK. Ativa automaticamente quando o código importa anthropic ou @anthropic-ai/sdk.

Exemplo clássico de invocação automática: você abre um arquivo que importa o SDK Anthropic e o Claude já sabe a documentação da API sem você precisar pedir.

🕹️ Modos de Invocação

Dois campos de frontmatter controlam quem pode acionar uma skill. Entendê-los é fundamental para criar skills que se comportam como você espera.

Matriz de Invocação

Frontmatter Você invoca Claude invoca Quando usar
(padrão) ✅ Sim ✅ Sim Skills de referência ou conhecimento que Claude deve aplicar quando relevante.
disable-model-invocation: true ✅ Sim ❌ Não Workflows com efeitos colaterais: /deploy, /commit, /send-slack. Você controla quando rodar.
user-invocable: false ❌ Não ✅ Sim Conhecimento de contexto que Claude usa internamente. Ex: legacy-system-context — não faz sentido como ação para o dev.

Skill manual: apenas você aciona

---
name: deploy
description: Deploy para produção
disable-model-invocation: true
---

Deploy $ARGUMENTS para produção:
1. Rode os testes
2. Build da aplicação
3. Push para o target
4. Verifique a saúde

O Claude nunca decide fazer um deploy por conta mesmo que o código pareça pronto.

Skill de contexto: apenas Claude usa

---
name: legacy-context
description: Contexto do sistema legado.
  Use quando editar módulos de
  pagamento ou integração com SAP.
user-invocable: false
---

O módulo de pagamento usa SOAP
porque o banco parceiro não suporta
REST. Não tente refatorar para HTTP
sem falar com o time de integração.

O Claude carrega esse contexto ao trabalhar em pagamentos. /legacy-context não aparece no menu.

🛠️ Criando sua Skill

Do zero a uma skill funcional compartilhada com o time em 4 passos.

1

Crie o diretório da skill

O nome do diretório vira o comando slash. review-pr/ → /review-pr.

mkdir -p .claude/skills/review-pr
2

Escreva o SKILL.md com frontmatter

A description é o campo mais importante — é ela que diz ao Claude quando usar a skill automaticamente.

---
name: review-pr
description: Revisa PRs com foco em segurança
  e performance. Use ao revisar um
  pull request ou diff de código.
disable-model-invocation: true
allowed-tools: Bash(gh *), Read
argument-hint: [número-do-pr]
---

Você é um revisor sênior deste projeto (NestJS + Prisma).
Analise o PR $ARGUMENTS verificando:

- Segurança: inputs sem validação, segredos expostos
- Performance: N+1 queries, loops custosos
- Testes: casos de borda descobertos

**Status**: ✅ Aprovado / ⚠️ Ajustes
**Issues**: por severidade (Alta / Média / Baixa)
3

Teste e refine

Execute com um PR real do time. Avalie se o output é preciso e específico para o seu stack. Itere 2–3 vezes na description e nas instruções até o resultado ser confiável.

4

Versione e compartilhe

Comite o diretório .claude/skills/. Ao clonar o repositório, todo dev já tem a skill disponível. Trate skills como documentação executável — atualize junto com o código.

git add .claude/skills/review-pr/
git commit -m "feat: add /review-pr skill"

🗃️ Sub-Skills & o Padrão Índice

Uma skill não precisa saber tudo de uma vez. Quando o domínio é grande — uma biblioteca UI, um sistema de design, uma API com dezenas de endpoints — carregar tudo no contexto seria desperdício. O Padrão Índice resolve isso: o SKILL.md vira um mapa leve que aponta para arquivos detalhados. O Claude carrega apenas o que a tarefa requer.

❌ Sem o padrão índice

.claude/skills/
├── button/SKILL.md      # 300 linhas
├── modal/SKILL.md       # 280 linhas
├── table/SKILL.md       # 420 linhas
├── data-grid/SKILL.md   # 500 linhas
├── form/SKILL.md        # 350 linhas
└── tooltip/SKILL.md     # 180 linhas

Todas as descriptions são carregadas no contexto ao iniciar a sessão. Com 6 skills detalhadas, você consome centenas de tokens antes de digitar uma linha.

✅ Com o padrão índice

.claude/skills/
└── ui-library/
    ├── SKILL.md         # ~30 linhas — índice
    ├── button.md        # carregado sob demanda
    ├── modal.md         # carregado sob demanda
    ├── table.md         # carregado sob demanda
    ├── data-grid.md     # carregado sob demanda
    ├── form.md          # carregado sob demanda
    └── tooltip.md       # carregado sob demanda

Apenas a description da skill ui-library entra no contexto inicial. O conteúdo detalhado de cada componente só é lido quando o Claude precisa.

Como funciona em runtime

1
Sessão inicia — apenas a description de ui-library é carregada: "Guia da biblioteca UI. Use ao trabalhar com Button, Modal, Table, DataGrid, Form, Tooltip."
2
Você pede: "Cria um DataGrid com seleção múltipla e ordenação por coluna"
3
Claude invoca a skill e lê o SKILL.md (índice). Identifica que a tarefa é sobre DataGrid → lê apenas data-grid.md.
4
Executa a tarefa com conhecimento completo sobre DataGrid — sem ter carregado Button, Modal, Form ou qualquer outro componente.

.claude/skills/ui-library/SKILL.md — o índice

---
name: ui-library
description: Guia completo da biblioteca de componentes UI
  deste projeto. Use ao criar, editar ou entender
  qualquer componente: Button, Modal, Table,
  DataGrid, Form ou Tooltip.
---

# Esta skill é um índice. Leia o arquivo do componente
# relevante antes de executar qualquer tarefa.

## Componentes disponíveis

| Componente | Arquivo          | Quando carregar                          |
|------------|-----------------|------------------------------------------|
| Button     | [button.md](button.md)     | Props, variantes, acessibilidade         |
| Modal      | [modal.md](modal.md)       | Portais, animações, foco trap            |
| Table      | [table.md](table.md)       | Ordenação, paginação, seleção de linhas  |
| DataGrid   | [data-grid.md](data-grid.md)  | Virtualização, edição inline, filtros    |
| Form       | [form.md](form.md)         | Validação, campos controlados, submit    |
| Tooltip    | [tooltip.md](tooltip.md)     | Posicionamento, delays, acessibilidade   |

## Instruções

1. Identifique qual componente é relevante para a tarefa
2. Leia o arquivo correspondente da tabela acima
3. Siga rigorosamente as convenções documentadas
4. Se a tarefa envolver múltiplos componentes, leia cada arquivo

# Nunca assuma convenções — sempre leia o arquivo antes.

.claude/skills/ui-library/data-grid.md — sub-skill detalhada

# DataGrid — Guia completo

## Instalação e import
import { DataGrid } from '@acme/ui/data-grid'

## Props obrigatórias
- rows: Row[]           — dados da grade
- columns: ColDef[]     — definição de colunas
- rowId: keyof Row      — campo único por linha

## Seleção múltipla
Ative com selectionMode="multiple".
Use onSelectionChange para capturar linhas selecionadas.
Não use selectedRows como state externo — o componente
gerencia internamente. Use selectedRowIds para seed inicial.

## Ordenação
Ative com sortable em cada ColDef.
Ordenação server-side: use onSortChange + externalSort=true.

## Padrão obrigatório: loading state
Sempre passe loading={isLoading}.
Nunca renderize DataGrid sem dados — use o esqueleto nativo.

## Anti-patterns conhecidos
- ❌ Não use índice de array como rowId — quebrará seleção
- ❌ Não mute rows diretamente — sempre passe novo array
💡

O ganho não é só tokens — é qualidade

Quando o Claude foca apenas no sub-skill relevante, ele lê o documento inteiro e com atenção — sem dividir atenção com 5 outros componentes que não têm nada a ver com a tarefa. O resultado é mais preciso, menos alucinação de props, e convenções seguidas corretamente.

🚀 Padrões Avançados

Recursos que ampliam o que skills conseguem fazer além de instruções simples.

A sintaxe !`comando` executa um shell command antes do conteúdo ser enviado ao Claude. O output substitui o placeholder — o Claude recebe dados reais, não o comando em si. 

---
name: pr-summary
description: Resume mudanças de um PR com contexto real
context: fork
agent: Explore
allowed-tools: Bash(gh *)
---

## Contexto do PR
- Diff: !`gh pr diff`
- Comentários: !`gh pr view --comments`
- Arquivos mudados: !`gh pr diff --name-only`

## Tarefa
Resuma este PR: o que mudou, por que,
e quais riscos existem.
Como funciona: Cada !`comando` executa imediatamente (pré-processamento). O output é injetado no prompt. O Claude recebe o resultado final com dados reais — não vê os comandos em si.

context: fork executa a skill em um subagente isolado. O conteúdo do SKILL.md vira o prompt que direciona o subagente, sem acesso ao histórico da conversa principal. 

---
name: deep-research
description: Pesquisa profunda em um tópico do codebase
context: fork
agent: Explore
---

Pesquise $ARGUMENTS em profundidade:

1. Use Glob e Grep para encontrar arquivos relevantes
2. Leia e analise o código
3. Resuma os achados com referências de arquivo
1. Contexto isolado

Subagente não tem acesso ao histórico da sessão principal.

2. Agent define ferramentas

agent: Explore → ferramentas de leitura otimizadas para exploração.

3. Resultado retorna

O output do subagente é resumido e devolvido para a conversa principal.

O diretório da skill pode ter arquivos de suporte. O SKILL.md referencia eles — o Claude os carrega somente quando necessário, mantendo o contexto enxuto. 

.claude/skills/review-pr/
├── SKILL.md           # instrução principal (máx. 500 linhas recomendado)
├── security-rules.md  # OWASP Top 10 detalhado — carregado quando necessário
├── examples.md        # exemplos de output esperado
└── scripts/
    └── fetch-pr.sh    # script executável — não carregado, executado

Referenciando em SKILL.md

⚠️ Limites & Cuidados

Skills mal configuradas criam comportamentos inesperados — especialmente com invocação automática.

❌ Description vaga → Claude usa quando não deve

description: Analisa código.

O Claude vai interpretar isso como relevante para qualquer coisa envolvendo código — a skill dispara em contextos que você não previa.

✅ Description com gatilhos específicos

description: Revisa PRs de APIs REST em busca de problemas de
  segurança e performance. Use ao revisar um pull request ou
  quando o usuário pedir para analisar um diff.

❌ Skill de deploy sem disable-model-invocation

O Claude pode decidir executar um deploy por conta própria ao detectar que "o código parece pronto". Workflows com efeitos colaterais irreversíveis sempre precisam de disable-model-invocation: true.

✅ Separar skills de conhecimento de skills de ação

Skills que informam (convenções, contexto legado) → deixe o Claude invocar. Skills que agem (deploy, commit, enviar mensagem) → disable-model-invocation: true.

❌ Skill com context: fork sem tarefa clara

Um context: fork em uma skill de guias de estilo ("use estas convenções…") faz o subagente receber diretrizes mas nenhuma tarefa acionável — retorna sem fazer nada.

✅ context: fork apenas para tarefas com instruções explícitas

Use fork para skills que têm uma tarefa clara: pesquisa, geração de relatório, execução de script. Para conhecimento de referência, use o modo padrão (inline).

🧠

Budget de contexto de skills

As descriptions das skills são carregadas no contexto para o Claude saber o que está disponível. Com muitas skills, elas podem exceder o budget (2% da janela de contexto, máx. ~16.000 chars). Execute /context para verificar se há skills sendo excluídas.
