📛 O Incidente Real

Isso não é hipotético. Durante nossas sessões, desenvolvedores deram autonomia total à IA, sem regras, sem limites, sem confirmações. O resultado foi destrutivo. Felizmente tínhamos backup. Mas o susto foi real.

🗄️

Incidente 1: Banco de Dados

Dev pediu: "limpa os dados de teste do banco". A IA interpretou como permissão irrestrita e executou DROP TABLE em múltiplas tabelas de produção sem nenhuma confirmação.

Tabelas perdidas: orders_2023, users_test, sessions
🗂️

Incidente 2: Arquivos do Projeto

Dev pediu: "reorganiza a estrutura do projeto". A IA deletou 47 arquivos que considerou "duplicados ou desnecessários", incluindo configs de CI/CD e migrations históricas.

Arquivos removidos: 47 · Recuperados do backup: 44

"O problema não foi a IA. Foi a ausência de limites."

A IA fez exatamente o que foi pedido de forma mais literal e abrangente do que o dev esperava. Sem regras explícitas, não havia como saber onde parar.

🖥️ Simulação: A mesma instrução, dois cenários

2. Essas tabelas têm dados relevantes em produção?

3. Prefiro usar TRUNCATE (preserva estrutura) em vez de DROP. Posso criar um script reversível.

Aguardando confirmação e esclarecimentos antes de executar qualquer ação.

📜 CONSTITUTION.md: As Regras Inegociáveis

A Constituição define o que a IA nunca pode fazer, o que precisa de confirmação, e os padrões obrigatórios de código e arquitetura. Ela é importada em toda conversa e serve como contrato entre o time e a IA.

Estas ações são proibidas sem confirmação explícita e textual do desenvolvedor:

✗

DROP TABLE / TRUNCATE / DELETE sem WHERE

Qualquer remoção de dados em massa

✗

git push --force

Em qualquer branch, incluindo feature branches

✗

rm -rf / deletar arquivos sem backup

Exclusão permanente de arquivos do projeto

✗

Alterar .env de produção

Variáveis de ambiente de produção são intocáveis

✗

Expor secrets / tokens em código

Hardcoded API keys, senhas ou tokens

✗

DROP DATABASE / operações globais de schema

Destruição de bases inteiras

Antes de executar estas ações, a IA deve pausar e obter confirmação:

?

Alterar schema de banco de dados

Adicionar/remover colunas, índices ou constraints

?

Instalar dependências globais

npm install -g, brew install, apt-get

?

Criar arquivos fora do escopo do plan.md

Qualquer arquivo não listado no plano aprovado

?

Modificar migrations existentes

Migrations já aplicadas não devem ser alteradas

?

Alterar pipelines de CI/CD

.github/workflows, Dockerfile, scripts de deploy

?

Alterar configurações de autenticação

Qualquer mudança no fluxo de auth/autorização

Padrões que se aplicam a todo código novo, sem exceção:

✓

TypeScript strict mode

Sem any implícito. Sem atalhos de tipagem.

✓

Testes para todo código novo

Unitários para lógica de negócio, integração para fluxos críticos

✓

Nomes em inglês, comentários em PT-BR

Variáveis, funções e arquivos em inglês

✓

Sem console.log em produção

Use o logger configurado no projeto

✓

Variáveis de ambiente via process.env

Nunca hardcoded. Sempre documentadas no .env.example

✓

Migrations reversíveis obrigatórias

Todo up() deve ter um down() funcional

Decisões arquiteturais que a IA não pode contornar ou ignorar:

⬡

Feature Folders

Código organizado por domínio, não por tipo de arquivo

⬡

Separação Repository / Service / Controller

Sem lógica de negócio em controllers ou rotas

⬡

Erros tipados, não strings

Use classes de erro customizadas, não throw "mensagem"

⬡

Dependências no TECH_STACK.md aprovadas

Não instale libs fora do inventário aprovado sem discussão

# CONSTITUTION.md: Regras Inegociáveis do Projeto
# ⚠️ Este arquivo é importado automaticamente em toda conversa.

## 🔴 NUNCA FAÇA (sem confirmação explícita e textual)
- DROP TABLE, TRUNCATE ou DELETE sem cláusula WHERE
- git push --force em qualquer branch
- rm -rf ou exclusão permanente de arquivos sem backup confirmado
- Alterar arquivos .env de produção ou staging
- Hardcodar secrets, tokens ou senhas em código
- DROP DATABASE ou operações destrutivas de schema global

## 🟡 PERGUNTE ANTES DE EXECUTAR
- Qualquer alteração de schema de banco (ADD/DROP COLUMN, índices)
- Instalar dependências globais no sistema
- Criar arquivos fora do escopo definido no plan.md
- Modificar migrations já aplicadas
- Alterar pipelines de CI/CD ou Dockerfiles

## ✅ PADRÕES OBRIGATÓRIOS (todo código novo)
- TypeScript strict mode sem `any` implícito
- Testes unitários para lógica de negócio
- Nomes de variáveis e funções em inglês
- Comentários e documentação em Português Brasileiro
- Sem console.log em código de produção
- Migrations sempre reversíveis (up + down)

## 🏗️ ARQUITETURA (não negocie sem aprovação do time)
- Feature Folders: organização por domínio, não por tipo
- Separação obrigatória: Repository / Service / Controller
- Erros tipados com classes customizadas
- Novas dependências somente se aprovadas no TECH_STACK.md

🗺️ CLAUDE.md como Índice, Não como Dump

O CLAUDE.md é carregado em toda conversa com o Claude Code. Cada linha desnecessária infla o contexto, aumenta o custo e distrai o modelo. A solução: transformá-lo em um roteador inteligente.

❌ Antipadrão: Dump de regras

CLAUDE.md com 400+ linhas de regras inline. Carregado inteiro em toda conversa.

# CLAUDE.md (400 linhas)

## Regras de TypeScript

Sempre use strict mode...

Nunca use any...

[... 50 linhas de TypeScript ...]

## Regras de Banco de Dados

[... 80 linhas de SQL rules ...]

## Arquitetura

[... 120 linhas ...]

→ 400 linhas consumidas em TODO contexto

✅ Correto: Roteador inteligente

CLAUDE.md com ~20 linhas. Importa apenas o essencial, roteia o resto.

# CLAUDE.md (~20 linhas)

@.claude/rules/CONSTITUTION.md

Para demais contextos, consulte:

.claude/rules/README.md

Carregue sob demanda:

- Banco → TECH_STACK.md

- Feature → ARCHITECTURE.md

→ Apenas CONSTITUTION.md sempre presente

⚙️ Hooks & Permissões: A Segunda Linha de Defesa

A CONSTITUTION.md instrui a intenção. Os hooks bloqueiam na execução. São camadas complementares, não substitutas. O .claude/settings.json é onde você configura ambos.

📜 CONSTITUTION.md

Contexto da IA. A IA sabe que não deve fazer X, isso influencia o raciocínio antes de propor uma ação. Pode ser ignorado se instruído explicitamente.

⚙️ Hooks no settings.json

Execução real. O hook intercepta antes do comando rodar e pode bloqueá-lo com exit 2. Não depende da IA, é uma barreira técnica.

⚙️ demo/.claude/settings.json: hook de segurança 
{
  "_demo_notice": "DEMO: Engineering Hour Part 6: adapte antes de usar em produção.",

  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{
        "type": "command",
        "command": "echo \"$CLAUDE_TOOL_INPUT\" | grep -iEq 'drop table|drop database|truncate|delete from|--force|rm -rf' && echo 'BLOQUEADO: comando destrutivo detectado pela CONSTITUTION' >&2 && exit 2 || exit 0"
      }]
    }],

    "PostToolUse": [{
      "matcher": "Bash",
      "hooks": [{
        "type": "command",
        "command": "echo 'Comando executado, verifique o resultado antes de continuar' >&2; exit 0"
      }]
    }]
  }
}

🔒 Permissões no settings.json

Além dos hooks, o Claude Code permite negar ferramentas específicas via permissions.deny:

"permissions": {
  "deny": [
    "Bash(rm:*)",         // bloqueia rm completamente
    "Bash(git push:*)",  // bloqueia git push direto
    "WebFetch(*)"           // bloqueia fetch externo se necessário
  ]
}

🧩 Segurança em MCP, Skills & Comandos

Cada camada de extensão do Claude Code traz benefícios e novos vetores de risco. A CONSTITUTION.md deve cobrir todas elas explicitamente.

Camada Mecanismo Risco principal Proteção recomendada
CONSTITUTION.md Contexto / instrução Ações não intencionadas Regras claras por categoria
settings.json → hooks Bloqueio na execução Comandos destrutivos PreToolUse com regex
MCP Servers Acesso a sistemas externos Credenciais de produção / PII Read-only + staging apenas
.claude/commands/ Skills customizadas Automação excessiva Revisão periódica + sem ações irreversíveis

🌐 MCP: Regras de Segurança

❌ Configuração perigosa

"postgres": {
  "args": [
    "postgresql://admin:senha@prod-db/app"
  ]
}
// Credenciais de produção + write access

✅ Configuração segura

"postgres": {
  "args": [
    "postgresql://readonly:*@staging-db/app"
  ]
}
// Staging + usuário read-only
Regra para CONSTITUTION.md: "Servidores MCP só podem usar credenciais de staging com permissão de leitura. Nunca conectar a produção via MCP em ambiente local."

⚠️ Limites & Cuidados

Guardrails bem configurados reduzem riscos, mas não os eliminam. Entender os limites é tão importante quanto implementar as proteções.

🗣️

Instrução ambígua supera qualquer regra

"Limpa tudo" é perigoso independentemente de quantas regras existam. A CONSTITUTION.md protege contra ações não intencionadas, não contra instruções mal formuladas pelo dev. A responsabilidade de ser claro é sempre do humano.

🤝

A IA pode ser convencida a ignorar regras

Se o dev disser "ignore a CONSTITUTION.md para este caso", a IA provavelmente vai obedecer. Guardrails de contexto não são barreiras técnicas, são instruções. Para bloqueios reais, use hooks no settings.json.

⚔️

Regras conflitantes paralisam ou confundem

Se a CONSTITUTION.md diz "nunca instale dependências" e o CLAUDE.md diz "otimize o ambiente", a IA vai flutuar. Mantenha os arquivos coerentes e revise conflitos sempre que uma regra for alterada.

🦠

Regras desatualizadas são perigosas

Uma CONSTITUTION.md de 6 meses pode proibir práticas que o time já adotou, ou não cobrir riscos novos introduzidos por novas ferramentas. Regras obsoletas geram falsa sensação de segurança.

🤖

Guardrails não substituem revisão humana

A IA pode gerar código estruturalmente correto e dentro de todas as regras, mas que resolve o problema errado. O julgamento de intenção ainda é 100% humano.

O Mantra da Parte 6:

"Guardrails não tornam a IA segura. Eles tornam os seus limites explícitos e explícito é sempre melhor que implícito."
