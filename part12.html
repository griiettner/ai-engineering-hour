🏗️ Sem arquitetura operacional, agente bom vira improviso caro

O problema não é conseguir uma boa resposta uma vez. O problema é conseguir a mesma qualidade na próxima task, em outra ferramenta, com menos contexto, menos drift e menos retrabalho. É isso que uma arquitetura operacional resolve.

Tese da sessão

Um sistema de agentes escalável começa por roteadores curtos, memória organizada, skills modulares, tasks explícitas e metadados regeneráveis.

Quando está bem montado

  • `AGENT.md` só roteia contexto, não tenta carregar o projeto inteiro
  • A mesma base serve Claude, Codex e Cursor sem duplicar a verdade
  • Cada task preserva sua história operacional dentro da própria pasta
  • Índices e stubs podem ser regenerados sem medo

Quando está mal montado

  • Prompt gigante tenta substituir memória, regras, tasks e tooling
  • Cada ferramenta recebe uma cópia diferente dos mesmos fluxos
  • Histórico de execução se mistura com conhecimento durável do projeto
  • Os arquivos de descoberta envelhecem mais rápido que o código

Bootstrap mínimo de entrada

O papel do arquivo inicial é mandar o agente para a rota certa com o menor custo possível.

# Contexto do Agente

Este arquivo é o contexto inicial leve para agentes.
Use-o como mapa de rotas, não como memória completa do projeto.

- Trabalho de task ou ticket: leia `.agents/tasks/index.yaml` primeiro
- Regras de workflow de tickets: leia `.agents/skills/tkt-management/SKILL.md`
- Roteador de memória durável: leia `.agents/memory/MEMORY.md`
- Contexto de arquitetura: leia `.agents/memory/architecture/index.yaml` primeiro
- Decisões duráveis: leia `.agents/memory/decisions/index.yaml` primeiro

Carregue apenas os arquivos relevantes para a tarefa atual.

🧭 Canônico + adapters é a decisão que segura o sistema

Se a sua fonte de verdade muda por ferramenta, a manutenção explode. O layout canônico vive em `.agents/`. O resto existe para compatibilidade, descoberta ou UX de cada ambiente.

Fonte de verdade

`.agents/`

Memória, skills, tasks e tooling regenerável vivem aqui. É o que você edita de verdade.

Compatibilidade

`.claude/skills/` e afins

Stubs, metadata local e bridges existem para a ferramenta consumir sem duplicar a lógica real.

Risco principal

Drift silencioso

Quando o time edita o adapter e esquece o canônico, o comportamento diverge sem ninguém perceber.

Claude Code · bridge explícita Codex · metadata local opcional Cursor · convenção e naming

Regra operacional

Edite o artefato canônico. Regenere o que for derivado. Nunca trate bridge como fonte da verdade.

🌲 A árvore precisa refletir função, não só organização estética

A estrutura recomendada separa bootstrap, memória, skills, tasks e tooling. Isso reduz custo de descoberta e deixa claro onde cada tipo de informação deve viver.

Camadas permanentes

`memory`, `skills` e `tasks` formam o núcleo. São eles que preservam contexto, procedimento e execução ao longo do tempo.

Camadas derivadas

`index.yaml`, shards e stubs ajudam descoberta. Eles só são confiáveis quando podem ser reconstruídos pelo repo.

🧠 Memory, skills e tasks têm ciclos de vida diferentes

Misturar esses três domínios é uma das formas mais rápidas de degradar um sistema de agentes. Cada um existe para uma pergunta diferente.

Memory

Conhecimento durável: arquitetura, princípios, ADRs, feedbacks e coisas que precisam sobreviver à task atual.

Skills

Procedimento reutilizável. A skill ensina como operar, não armazena a história da última execução.

Tasks

História operacional por work item: objetivo, plano, updates, evidência e relatório de conclusão.

🔌 Cada ferramenta descobre contexto de um jeito diferente

O erro comum é fingir simetria. Claude, Codex e Cursor têm capacidades diferentes de descoberta, invocation e metadata. A arquitetura boa lida com isso explicitamente.

Claude Code

Bridge explícita

Use `.claude/skills/` como stub gerado e mantenha `CLAUDE.md` curto, apontando de volta para a estrutura canônica do repo.

Codex

Metadata local opcional

Quando fizer sentido, mantenha `agents/openai.yaml` ao lado da skill canônica para display metadata e política de invocation.

Cursor

Convenção primeiro

Sem adapter nativo forte, padronize naming, argumentos e contratos de uso. O objetivo é preservar o workflow, não simular features inexistentes.

🗃️ Índices só funcionam se forem baratos de ler e baratos de reconstruir

O papel de `index.yaml` não é virar banco de dados paralelo. É reduzir custo de descoberta. Se ele não acompanha a árvore real, vira ruído e engana o agente.

Regra operacional

Se um índice é importante para o agente, ele precisa ser gerado e validado pelo próprio repositório.

Índice temático de memória

entries:
  - id: principles-constitution
    file: .agents/memory/principles/constitution.md
    status: ativo
  - id: architecture-overview
    file: .agents/memory/architecture/overview.md
    status: ativo

Shard de tasks por intervalo

- range: TKT-001-025
  file: .agents/tasks/indexes/TKT-001-025.yaml
  count: 2
  updated: 2026-05-25
  statuses:
    - concluido: 1
    - em_andamento: 1

Esquema estável

Campos previsíveis em YAML evitam parse ambíguo e facilitam automação simples.

Sync e validação

Scripts como `tkt_sync.py`, `tkt_validate.py` e `sync_claude_skills.py` mantêm tasks, índices e bridges alinhados.

Descoberta barata

Leia o menor arquivo útil primeiro. O índice existe para evitar varrer a árvore inteira.

⚠️ Os erros mais caros não são técnicos, são estruturais

Quando a arquitetura operacional quebra, o efeito aparece como contexto inchado, task sem rastreabilidade e comportamento inconsistente entre ferramentas.

Transformar `AGENT.md` em dump file

Se o bootstrap tenta ensinar tudo de uma vez, ele deixa de ser roteador e vira custo fixo de contexto.

Editar a cópia exposta em vez da skill canônica

Isso cria drift entre `.agents/skills/` e o adapter da ferramenta, e ninguém sabe mais qual versão vale.

Misturar memória durável com histórico de task

Conhecimento de projeto e narrativa operacional têm ciclos de vida diferentes e precisam de casas diferentes.

Assumir descoberta igual em todas as ferramentas

Claude, Codex e Cursor não têm o mesmo modelo de invocation, então a integração não pode ser tratada como se fosse idêntica.

Manter tudo regenerável

O que for derivado deve ser recriado por script. O que for canônico deve ser pequeno, explícito e fácil de editar.

🚀 O caminho curto é começar do scaffold certo

Em vez de inventar a árvore inteira no improviso, o fluxo recomendado é clonar `griiettner/agents-scaffolding`, trocar placeholders e adaptar o conteúdo canônico ao seu projeto.

Clone direto

git clone https://github.com/griiettner/agents-scaffolding
cd agents-scaffolding
1

Clone o scaffold

Use a árvore pronta como base para evitar discutir naming, indexação e bridges a cada repo novo.

2

Troque placeholders por regras reais

Atualize `AGENT.md`, `CLAUDE.md`, arquitetura, princípios, decisões e exemplos de task para refletir o projeto verdadeiro.

3

Defina skills canônicas

Cada workflow recorrente deve ganhar sua própria skill em `.agents/skills/`, e não um bloco perdido em um prompt gigante.

4

Regenere adapters e índices

Sempre que a estrutura mudar, rode sync e validação para manter bridges e metadata coerentes.

5

Abra a ferramenta na raiz do repo

É isso que permite descoberta local de memória, skills e tasks sem depender de contexto manual em toda sessão.

Resumo final

Trate `.agents/` como sistema de registro, exponha compatibilidade por ferramenta só onde necessário e mantenha tudo que for derivado sempre regenerável.
